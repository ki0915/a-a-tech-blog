Secondary Hacking Response

Overview

A secondary hacking incident occurred on the same server that we previously responded to. It seems that my earlier concerns were valid; the server remained vulnerable due to the lack of VPN usage. Fortunately, the databases and procedures remained intact. This document will outline the response and the steps taken to reset the server.

Risk Identification

The computer that was hacked again had its password changed. To resolve this, I followed the steps outlined in the 1st Hacking Response to reset the locked password and gain access. Upon investigation, I discovered that a virtual machine was running on the server using VMware, and it was being used to mine cryptocurrency.

Identified Risks

  1. Forced installation of a virtual machine on the server for cryptocurrency mining.
  2. Network settings were altered.
  3. Password was changed.

Risk Mitigation

While I initially planned to reset the server and implement VPN configurations, the client requested a migration to AWS Cloud. Before temporarily formatting the physical server, I conducted a root cause analysis. The server was then shut down to prevent further damage.

Risk Elimination

The connection between the server and the network was severed to prevent the virus or hacker from potentially accessing other entities.

Recovery and Restoration

Although I have certifications and experience with Azure, I am less familiar with AWS. To address this, I familiarized myself with the basics of AWS configurations and operations before proceeding with the recovery.

Instance Configuration

  • Created an instance on the AWS service owned by the client to host the service being recovered.

Settings Location: Seoul Instance Image: Windows Server Instance Type: Cost-effective paid option IP Address: Determined as per the client’s requirements

Security Configuration

Inbound Rules: Only essential service ports are accessible. Source: Access limited to specific IPs

Detailed configurations are not mentioned here.

Server Setup

  • Set up the IIS server as per the previous configuration.

Database Setup

  • Configured MSSQL using AWS RDS.
  • Detailed server configurations are not mentioned here.

Restoring Backed-Up Database

  • Connected to the database using SSMS and performed a restoration of the backed-up database.
Problem Encountered
  • During the database restoration in MSSQL, the process halted, and not all data was restored.
  • This issue persisted even when using SQL CMD instead of SSMS.
Root Cause Identified
  • The issue was caused by malicious binary image files inserted into the database from the CMS server.
  • The excessively large image files exceeded the memory allocated to MSSQL, causing it to halt.
  • This revealed that one of the hacking methods involved a file injection attack.
Solution
  • I used Windows commands to split the SQL file, removed the malicious images, and then performed the backup again.
  • The server was successfully restored following this process.

Conclusion

One of the attack vectors used against the server was a file injection attack. This occurred because the server logic failed to verify the integrity of uploaded files and text. The files were uploaded as binary code, which the server did not adequately filter. As a result, I strengthened the integrity verification logic in the server and redeployed it. This incident reinforced the critical importance of security.
