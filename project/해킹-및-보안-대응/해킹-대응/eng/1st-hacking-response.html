1st Hacking Response

While developing servers at the company, I heard news that an external server had been hacked. Since it was just a CMS server managing specific contents externally, it wasn't a massive issue, but quick recovery was essential for content management. As someone with a background in security and ongoing study, I started addressing the problem.

  • Having recently obtained CISA Associate, I will attempt to write this according to the incident response process.
  • However, I had not studied CISA itself at that time, so there may be discrepancies with the process.

Risk Identification

Plan for Risk
  1. Collect host computer information (briefly conducted as information was already available)
  2. Login to server/DB host computers
  3. Check server and DB status
  4. Search for malicious files and processes

Collecting Host Computer Information

  • Detailed information not provided, but collected operating system information, etc.

Login to Server/DB Host Computers

Server login failed
  • It was found that the admin account's password had been changed by the hacker.
  • However, the guest account password remained intact. ** Since guest account login was possible, I decided to use it to exploit the vulnerability known as privilege escalation.

Privilege escalation occurs when a user or process gains more privileges than intended. In this case, access control failed, allowing me to utilize this vulnerability. I should note that these actions were taken to recover the server and were agreed upon by the company.

1. Login with the guest account
2. Run CMD as administrator
   CMD:\> net user Administrator new_password
3. Login with the admin account

This process was successful as planned.

Success

It was confirmed that I successfully accessed the server computer. Permissions were not configured as I had expected, but this only applied to the server computer and not the DB computer.

DB login failed

It was found that even the guest account for the DB had been changed. In this case, I needed to find another method instead of the previous approach. I needed to prepare the following:

Preparation: Windows USB same version as the host computer.

Step 1: Create Windows Server 2012 installation media on an accessible computer.
Step 2: Insert it into the locked Windows Server 2012 computer and repeatedly press the boot key to enter BIOS. Set the installation disk as the first boot option, then press **F10** to save and exit.
Step 3: Enter language and other basic settings, then click **Next**.
Step 4: Click **Computer Recovery** in the bottom left corner.
Step 5: Click **Troubleshoot** in the **Choose an option** window.
Step 6: Click **Command Prompt** in the **Advanced options** window.
Step 7: Enter the following commands, pressing **Enter** at the end of each line.

1. D:
2. CD Windows\system32
3. Ren utilman.exe utilman.exe.bak
4. Ren cmd.exe utilman.exe
5. WPEUtil reboot

The computer will automatically restart.

*** However, Step 7 did not work for me because CMD itself stated that I did not have permission to change the password. In that case, try replacing cmd.exe with netplwiz.exe in the command. Alternatively, executing that process from CMD can easily change the password.
Successful Access

I was able to access through this method.

In-depth Problem Exploration
  • Successfully identified the current malicious processes and locations of malicious files.
  • Both the server and DB were attacked, and the DB contained a dormant SQL-related virus.

Threat Control

  • Isolated the server/DB environment under threat, including malicious code and processes.
  • Also ensured continuity by performing backups and continuity assurance tasks.

Threat Removal

  • Deleted all malicious code and processes.
  • Blocked access to problematic paths to prevent further breaches.

Recovery and Restoration

  • As no data was lost or damaged, only the recovery of the server and DB hosts was conducted.
  • Redeployed any processes that were damaged in the process.

Feedback and Clarification

  • Upon reviewing the hacking incident, it was determined that it occurred due to external access to servers and DB computers connected to the internal network.
  • While external access is necessary for user access, leaving it as it was posed a significant risk.
  • Therefore, I proposed enhancing server confidentiality through VPN access, but the company rejected this and opted to restrict access to specific IP addresses for the server and its web applications.

This incident was resolved for now, but issues still remained. Finding a brute-force script text file left by the hacking script suggested potential server breach via brute force or vulnerabilities in web applications. From this point, I focused more on studying security to effectively respond to such incidents.
