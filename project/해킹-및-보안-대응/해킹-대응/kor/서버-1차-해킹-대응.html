1차 해킹 대응

회사에서 서버를 개발하던 중에 외부에 있는 서버가 해킹 당했다는 소식을 들었다.
단순히 외부에서 특정 컨텐츠들을 관리해주는 CMS 서버가 전부였기 때문에 어마어마한 문제는 아니었지만, 컨텐츠 관리 위해서는 빠른 복구가 필수였다. 마침 보안을 전공하였고 공부도 꾸준히 하고 있었던 나는 문제를 대응하기 시작했다.

  • 필자가 현재 CISA Associate를 취득하였으니 사고 대응 프로세스에 맞춰 글을 작성해보겠다.
  • 하지만, 그 당시에는 CISA 공부 자체를 한 적이 없기에 프로세스와 안맞는 부분이 있을 수 있다.

위험 식별

위험을 위한 계획
  1. 호스트 컴퓨터 정보 수집(이건 이미 정보가 나와있기에 간략하게 진행함)
  2. 서버/DB 호스트 컴퓨터 로그인
  3. 서버와 DB 상태 확인
  4. 악성 파일과 프로세스 탐색

호스트 컴퓨터 정보 수집

  • 자세한 정보는 기술하지 않겠지만 운영 체제 등의 정보 수집

서버 / DB 호스트 컴퓨터 로그인

서버 로그인 실패
  • 확인 결과 해커에 의해 admin의 계정 비밀 번호가 변경되었다.
  • 하지만, 게스트 계정의 비밀번호는 유지되고 있었다.
    ** 게스트 계정 로그인은 가능하니 이를 통하여 보안 취약점 중 하나인 권한 에스컬레이션을 사용하기로 했다.

권한 에스컬레이션이란 사용자나 프로세스에 의도한 권한보다 더 많이 부여한 경우 발생한다. 이 경우는 접근 통제에 실패하였기에 필자가 취약점을 이용할 수 있었다.
 미리 말해두지만, 서버 복구 위해서 수행한 것이기에 회사에 합의가 된 사항이다. 

1. 게스트 계정으로 접속
2. 관리자 권한으로 CMD 실행
	CMD:\> net user Administrator 원하는 비밀번호
3. 어드민 계정으로 접속

이 과정을 수행하였다. 원리는 간단하다.

게스트 계정에서 어드민 계정의 비번을 설정할 수 있기에 그걸 이용하여 어드민 계정의 비번을 수정하여 호스트에 접속하는 것이다.

성공

확인 결과 성공적으로 서버 컴퓨터에 접속하였다. 필자가 생각한 대로 권한 설정이 되어 있지 않았다. 하지만 이건 서버 컴퓨터에만 적용된 사항일 뿐 DB 컴퓨터는 적용되지 않았다.

DB 로그인 실패

확인 결과 DB는 게스트 계정마저 바뀌어 있었다. 이 경우에는 위와 같은 방법 대신에 다른 방법을 사용해야 한다.
필자는 이를 이루기 위해 다른 방법을 고안해야 했다.
이를 위한 준비물은 아래와 같다.

준비물: 호스트 컴퓨터 버전과 같은 윈도우 USB

1 단계  액세스 가능한 컴퓨터에 Windows Server 2012 설치 미디어를 만듭니다.
2 단계 : 잠긴 Windows Server 2012 컴퓨터에 삽입하고 부팅 키를 계속 눌러 BIOS로 들어갑니다. 설치 디스크를 첫 번째 부팅 옵션으로 설정한 다음 **F10** 키를 눌러 저장하고 종료합니다.
3 단계 : 언어 및 기타 기본 설정을 입력하고 **다음** 버튼을 클릭합니다.
4 단계 : 그런 다음 왼쪽 하단 모서리에 있는 **컴퓨터 복구**를 클릭합니다.
5 단계 : **Choose an option(옵션 선택**) 창에서 **Troubleshoot(문제 해결**)을 누릅니다.
6 단계 : **Advanced options(고급 옵션**) 창이 표시되면 **Command Prompt(명령 프롬프트)를** 클릭합니다.
7 단계 : 다음 명령을 입력하고 각 줄의 끝에서 **Enter** 키를 누릅니다.

1. 1. 디:
2. 2. CD 윈도우\system32
3. 3. 렌 utilman.exe utilman.exe.bak
4. 4. 렌 cmd.exe utilman.exe
5. 5. WPEUtil 재부팅

그러면 컴퓨터가 자동으로 다시 시작됩니다.

*** 하지만 저는 이 7단계 방법이 통하지 않았습니다. cmd 자체에서 비밀번호를 바꿀 권한이 없다고 떴기 때문입니다. 그럴 때에는 cmd.exe 를 넣어야 하는 부분에 netplwiz.exe 를 넣어보세요. 그게 아니어도 cmd 에서 저 프로세스를 실행하면 손쉽게 비밀번호를 바꿀 수 있습니다.
접속 성공

이를 통하여 접속에 성공할 수 있었습니다.

본격적인 문제 탐색
  • 현재 실행되고 있는 악성 프로세스와 악성 파일들의 위치를 파악하고 어떠한 경로로 들어왔는지 파악성공
  • 서버와 DB 모두 공격 받았으면 DB에는 SQL 관련 바이러스가 잠복해있었다.

위협 통제

  • 현재 위협받고 있는 서버/DB 환경과 악성 코드와 프로세스를 격리하였습니다.
  • 그리고 연속성을 확보해주기 위하여 백업과 연속성 확보 작업을 진행하였습니다.

위협 제거

  • 악성 코드와 프로세스를 전부 삭제하였습니다.
  • 문제가 되는 경로를 통한 접근이 불가능하도록 해당 경로를 차단하였습니다.

복구 및 복원

  • 사라지거나 망가진 데이터는 없었기에 서버와 DB 호스트 자체에 대한 복원만 실시하였습니다.
  • 그 과정에서 손상된 프로세스 다시 재배포하였습니다.

피드백 및 구체화

  • 해킹 사고를 검토해본 결과, 내부망과 연결되어 있는 서버 및 DB 컴퓨터를 외부의 어떤 개체도 접근이 가능하였기에 발생한 것이었습니다.
  • 사용자들이 외부에서 접근하기에 외부망을 뚫어두어야 하지만, 이 상태로 그냥 두는 것은 위험하다고 판단 하였습니다.
  • 그렇기에 VPN을 통한 접속으로 서버의 기밀성을 보완하자 제안하였습니다.
  • 하지만 회사에서 이를 거부하였기에 특정한 아이피만 서버와 서버의 웹 애플리케이션에 접속 가능하록 조치하였습니다.

이를 통하여 일단 문제는 일단락되었지만, 아직 문제는 여전히 남아있었다. 해킹 스크립트에 부르트 포스용 텍스트 파일이 남아있는 걸 보면 컴퓨터 자체는 부르트 포스로 뚫렸거나 웹 애플리케이션 취약점을 통한 해킹일 가능성이 높았다.

이를 기점으로 좀 더 보안쪽 공부에 더 공을 기울였던 것 같다. 이런 일에 효율적으로 대응하기 위해서다.
