Web Browser

  • The web browser is a convenient tool that can be easily used on any system. There are various ways to collect information about a target using a web browser.
  • The web browser accesses websites as follows:

When accessing a website via HTTP, it typically uses TCP port 80. When accessing a website via HTTPS, it typically uses TCP port 443.

While browsing a page, you can open the developer tools in Firefox by pressing Option + Command + I ( ) on a PC or Mac. You can also use a similar shortcut ⌥ + ⌘ + I to start Chrome or Chromium. The developer tools allow you to inspect many items received by the browser and exchanged with the remote server. For example, you can view and modify JavaScript (JS) files, inspect cookies set on the system, and explore the folder structure of site content.

There are also many add-ons for Firefox and Chrome that can assist in penetration testing. Here are some examples:

Example
  • FoxyProxy allows you to quickly change the proxy server used to access the target website. This browser extension is convenient if you regularly need to switch proxy servers or use tools like Burp Suite. You can get FoxyProxy for Firefox here.
  • User-Agent Switcher and Manager provides the ability to masquerade as if you are accessing a web page from a different operating system or browser. For instance, you can pretend to be browsing a site using an iPhone even though you are actually using Mozilla Firefox. You can download User-Agent Switcher and Manager for Firefox here.
  • Wappalyzer provides insights into the technologies used on the visited website. This extension is especially handy for collecting all this information while browsing the site like any other user. You can find Wappalyzer for Firefox here.

Ping

Ping is reminiscent of a ping pong game. You throw the ball and expect it to come back. The primary purpose of ping is to check whether a remote system can connect back to the user, i.e., it was originally used to verify network connectivity. However, we are more interested in the various uses of ping to verify if the remote system is online.

In simple terms, the ping command sends packets to a remote system, and the remote system responds. This allows you to conclude that the remote system is online and that the network is functioning between the two systems.

You can use ping on the AttackBox terminal as ping MACHINE_IP or ping HOSTNAME. In the latter case, the system will resolve HOSTNAME to an IP address before sending ping packets. On a Linux system, if you don't specify a count, you must press CTRL+c to forcibly stop it. Therefore, you might consider sending only 10 packets with ping -c 10 MACHINE_IP. This is the same as ping -n 10 MACHINE_IP on MS Windows systems.

Technically speaking, ping belongs to the ICMP (Internet Control Message Protocol).

Using ping, we can determine whether communication with the host is normal and if the PC is operational.

If you do not receive a ping response, there are several explanations for why you might not have received it. For example:

Example
  • The target computer is not responding. It might still be booting, turned off, or the OS may have crashed.
  • It has been disconnected from the network, or a network device along the route is faulty.
  • A firewall is configured to block such packets. The firewall could be software running on the system itself or a separate network appliance. By default, the MS Windows firewall blocks ping.
  • The system has been disconnected from the network.

Traceroute

The traceroute command tracks the path that packets take from your system to another host. The purpose of traceroute is to find the IP addresses of the routers or hops that packets pass through as they travel from your system to the target host. This command also indicates the number of routers between the two systems. It is useful because it shows the number of hops (routers) between your system and the target host. However, since many routers use dynamic routing protocols that adapt to network changes, the path that packets take can change.

The command used on Linux and macOS is traceroute 10.10.132.220, and on MS Windows, it is tracert 10.10.132.220. These commands attempt to discover the routers on the path from your system to the target system.

However, when the TTL reaches 0, the packet is discarded, and an ICMP TTL (Time-to-Live) exceeded message is sent back to the original sender. In the next figure, the system sets TTL to 1 before sending the packet to the router. The first router on the path decreases the TTL by 1, making it 0. As a result, this router discards the packet and sends an ICMP time exceeded message. Some routers are configured not to send these ICMP messages when discarding packets.

On Linux, traceroute begins by sending a UDP datagram with a TTL of 1 inside an IP packet. Therefore, when the first router encounters TTL=0, it returns an ICMP TTL exceeded message. Thus, when TTL is 1, the IP address of the first router is revealed. Then it sends another packet with TTL=2. This packet is discarded at the second router, and so on. Let's try this on a live system.

In the following example, we run the same command on TryHackMe's AttackBox: traceroute tryhackme.com. If the execution is different, the path that the packets take may vary.

In the second execution of the traceroute program, we observe that the packet traveled a longer route, passing through 26 routers. When running traceroute on a system within a network, the route may not change. However, if the packet has to travel through external routers outside the network, the route cannot be expected to remain fixed.

In summary, we can observe the following:

  • The number of hops/routers between the system and the target system can vary depending on the time at which traceroute is executed. Even if you are on the same network or repeat the traceroute command within a short period, there is no guarantee that the packet will always follow the same route.
  • Some routers return public IP addresses. Depending on the scope of the intended penetration test, some of these routers may be inspected.
  • Some routers do not return a response.

Telnet

The TELNET (Teletype Network) protocol was developed in 1969 to communicate with remote systems through a command-line interface (CLI). Therefore, this command uses the TELNET protocol for remote management via Telnet. The default port used by Telnet is 23. From a security perspective, Telnet sends all data, including usernames and passwords, in plain text. Sending in plain text allows anyone with access to the communication channel to easily steal login credentials. A secure alternative is the SSH (Secure Shell) protocol.

However, since the Telnet client is simple, it can be used for other purposes. Knowing that the Telnet client relies on the TCP protocol, you can use Telnet to connect to any service and retrieve its banner. By using telnet 10.10.132.220 PORT, you can connect to any service running on TCP at 10.10.132.220 PORT, and you can exchange a few messages as long as encryption is not being used.

Let's assume you want to retrieve detailed information about a web server listening on port 80. You connect to the server on port 80 and then communicate using the HTTP protocol. There is no need to dive into the HTTP protocol; simply issue GET / HTTP/1.1. If you want to specify something other than the default index page, you can request GET /page.html HTTP/1.1. Additionally, you’ve specified that the remote web server uses HTTP version 1.1 for communication. To receive a valid response instead of an error, you must input some value for the host (e.g., host: example) and press Enter twice. Following these steps will result in the requested index page being served.

Netcat

Netcat, or simply nc, offers various applications that can be of great value to penetration testers. Netcat supports both the TCP and UDP protocols. It can operate as a client connecting to a listening port, or it can act as a server listening on a selected port. Therefore, it is a convenient tool that can be used as a simple client or server over TCP or UDP.

First, you can connect to a server and collect banners using nc 10.10.132.220 PORT, as you did with telnet 10.10.132.220 PORT. You might need to press SHIFT+ENTER after the GET line.

pentester@TryHackMe**$** nc 10.10.132.220 80 GET / HTTP/1.1 host: netcat HTTP/1.1 200 OK Server: nginx/1.6.2 Date: Tue, 17 Aug 2021 11:39:49 GMT Content-Type: text/html Content-Length: 867 Last-Modified: Tue, 17 Aug 2021 11:12:16 GMT Connection: keep-alive ETag: "611b9990-363" Accept-Ranges: bytes

In the terminal shown above, we used Netcat to connect to port 80 on MACHINE_IP with the command nc 10.10.132.220 80. We then executed a GET request for the default page using GET / HTTP/1.1. The client is specifying that the target server supports HTTP version 1.1. Finally, since you need to specify a name for the host, you enter host: netcat on a new line; the host name can be arbitrary as it won’t impact this exercise.

Based on the received output, you can see that Nginx version 1.6.2 is handling incoming connections on port 80, identified by Server: nginx/1.6.2.

Below are the command options:

Option Meaning
-l Listen mode
-p Specify port number
-n Numeric only; no host name resolution via DNS
-v Verbose output (optional but useful for troubleshooting)
-vv Very verbose (optional)
-k Continue listening after the client disconnects
  • The -p option must appear immediately before the port number to be received.
  • The -n option prevents DNS lookups and warnings.
  • Root privileges are required to receive port numbers below 1024.
