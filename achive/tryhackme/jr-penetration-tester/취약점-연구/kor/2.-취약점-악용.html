1.   자동화와 수동 취약점 조사

취약성 스캐너는 애플리케이션의 결함을 신속하게 조사할 수 있는 편리한 수단입니다.

예를 들어 취약점 스캐너 Nessus에는 무료(커뮤니티) 에디션과 상업용이 있습니다. 1년 라이센스에 수천 파운드의 비용이 드는 상업용 버전은 침투 테스트 서비스 또는 감사를 제공하는 조직에서 사용될 가능성이 높습니다. 

아래 표에 취약성 스캐너 사용의 장단점을 자세히 설명했습니다.
장점 단점
자동 스캔은 반복하기 쉽고 결과를 팀 내에서 쉽게 공유할 수 있습니다. 사람들은 종종 이러한 도구에 의존하게 될 수 있습니다.
이 스캐너는 빠르고 수많은 애플리케이션을 효율적으로 테스트할 수 있습니다. 그들은 매우 "시끄럽고" 많은 트래픽과 로깅을 생성합니다. 방화벽 등을 우회하려는 경우에는 좋지 않습니다.
오픈 소스 솔루션이 존재합니다. 오픈 소스 솔루션은 종종 기본적이며 유용한 기능을 갖추려면 값비싼 라이선스가 필요합니다.
자동화된 스캐너는 수동으로 검색하기 어려울 수 있는 다양한 취약성을 광범위하게 다룹니다. 그들은 종종 애플리케이션에서 모든 취약점을 찾지 못합니다.

Metasploit과 같은 프레임워크에는 종종 일부 모듈에 대한 취약점 스캐너가 있습니다.

취약성에 대한 수동 스캐닝은 종종 침투 테스터가 개별 애플리케이션이나 프로그램을 테스트할 때 선택하는 무기입니다. 실제로 수동 스캐닝은 동일한 취약점 검색을 포함하며 자동 스캐닝과 유사한 기술을 사용합니다.

궁극적으로 두 기술 모두 응용 프로그램이나 프로그램의 취약성 테스트를 포함합니다. 이러한 취약점에는 다음이 포함됩니다.
취약점 설명
보안 구성 오류 보안 구성 오류에는 개발자 감독으로 인한 취약성이 포함됩니다. 예를 들어, 애플리케이션과 공격자 간의 메시지에 서버 정보가 노출됩니다.
손상된 액세스 제어 이 취약점은 공격자가 다른 방법으로는 접근할 수 없는 응용 프로그램의 일부에 접근할 수 있을 때 발생합니다.
안전하지 않은 역직렬화 이것은 애플리케이션을 통해 전송되는 데이터의 안전하지 않은 처리입니다. 공격자는 응용 프로그램에 악성 코드를 전달하여 실행될 수 있습니다.
injection 공격자가 응용 프로그램에 악성 데이터를 입력할 수 있는 경우 삽입 취약점이 존재합니다. 이는 입력이 유해하지 않은지 확인하지 못하기 때문입니다.

질문:

귀하는 침투 테스트 기한이 임박하여 웹 애플리케이션을 신속하게 스캔해야 합니다. 자동 스캐너를 사용하시겠습니까? (예/아니)

답: Yay

웹 애플리케이션을 테스트하고 있으며 데이터베이스에서 데이터를 입력하고 검색할 수 있음을 발견했습니다. 이것은 어떤 취약점입니까?

답: Injection

다른 사용자를 가장하는 데 성공했습니다. 이것은 어떤 취약점입니까?

답: Broken Access Control

2. 수동 익스플로잇 찾기

래피드**7**

Exploit DB 및 NVD와 같은 다른 서비스와 마찬가지로 Rapid7은 취약점 연구 데이터베이스입니다.

또한 데이터베이스에는 널리 사용되는 Metasploit 도구를 사용하여 응용 프로그램을 악용하기 위한 지침이 포함되어 있습니다

GitHub

GitHub 는 소프트웨어 개발자를 위해 설계된 인기 있는 웹 서비스입니다. 이 사이트는 공동 작업이 가능하도록 응용 프로그램의 소스 코드를 호스팅하고 공유하는 데 사용됩니다. 그러나 보안 연구원들은 앞서 언급한 이유 때문에 이 플랫폼을 선택했습니다. 보안 연구원은 PoC(Proof of Concept)를 GitHub에 저장하고 공유하여 이를 익스플로잇 데이터베이스로 전환합니다.

GitHub는 누구나 계정을 만들고 업로드할 수 있기 때문에 희귀하거나 새로운 익스플로잇을 찾는 데 매우 유용합니다. 대체 익스플로잇 데이터베이스와 같은 공식적인 과정이 없습니다. 왜냐하면 누구나 업로드할 수 있기 때문이죠.  즉, 지원이 거의 또는 전혀 제공되지 않는 경우 PoC가 작동하지 않을 수 있다는 단점도 있습니다.

GitHub는 태깅 및 키워드 시스템을 사용합니다. 즉, "PoC", "취약성" 등과 같은 키워드로 GitHub를 검색할 수 있습니다. 글을 쓰는 시점에 "cve"라는 키워드가 있는 저장소가 9,682개 있습니다. 프로그래밍 언어별로 결과를 필터링할 수도 있습니다.

SearchSploit

Searchsploit은 Kali Linux와 같은 인기 있는 침투 테스트 배포판에서 사용할 수 있는 도구입니다. 

 이 도구는 시스템의 익스플로잇 복사본을 포함하는 Exploit-DB의 오프라인 복사본입니다. 

애플리케이션 이름 및/또는 취약점 유형별로 searchploit을 검색할 수 있습니다. 예를 들어, 아래 스니펫에서는 사용할 수 있는 Wordpress와 관련된 익스플로잇에 대해 searchsploit을 검색하고 있습니다. 다운로드할 필요가 없습니다!

질문:

개념 증명을 업로드하려는 경우 보안 연구원으로서 어떤 웹사이트를 사용하시겠습니까?

답: Github

인터넷에 연결되지 않은 사이트에서 침투 테스트를 수행하고 있습니다. 사용할 익스플로잇을 찾는 데 사용할 수 있는 도구는 무엇입니까?

답: Searchsploit

3.  수동 착취의 예

취약한 애플리케이션이나 서비스를 실행 중인 대상에서 명령을 실행할 수 있으면 이전에는 애플리케이션이나 서비스만으로는 수행할 수 없었던 파일을 읽거나 명령을 실행할 수 있습니다. 또한 이를 악용하여 기계에 발판을 마련할 수 있습니다. 발판은 취약한 시스템의 콘솔에 대한 액세스로, 여기서 네트워크의 다른 애플리케이션이나 시스템을 악용할 수 있습니다.

취약한 시스템에서 명령을 원격으로 실행할 수 있도록 익스플로잇을 사용하여 작업 2의 애플리케이션에서 원격 코드 실행을 수행할 것입니다.

시작하기 전에 익스플로잇은 거의 즉시 사용할 수 없으며 사용할 준비가 되어 있다는 점에 유의해야 합니다. 환경이나 대상에서 작동하기 전에 일부 구성이 필요한 경우가 많습니다. 구성 수준은 익스플로잇에 따라 달라지므로 애플리케이션에서 동일한 취약점에 대한 여러 익스플로잇을 발견하는 경우가 많습니다. 자신에게 가장 적합하거나 유용한 익스플로잇을 파악하는 것은 사용자의 몫입니다.

예를 들어, 아래 스니펫에서 공격하는 시스템의 IP 주소를 반영하도록 몇 가지 옵션이 변경된 것을 볼 수 있습니다.

익스플로잇 수정(이전)

nano exploit.py

mymachine="192.168.1.10"

port="1337"

익스플로잇 수정(이후)

nano exploit.py

mymachine="10.13.37.10"

port="1337"

익스플로잇을 올바르게 구성했으면 이 익스플로잇을 자세히 읽고 사용 방법을 이해해 봅시다. 아래 스니펫에서 익스플로잇을 실행할 때 두 가지 인수를 제공해야 함을 알 수 있습니다.

익스플로잇에 대한 인수 나열

exploit.py --help

To use this exploit, provide the following arguments:

-u The URL of the application

-c the command that you wish to execute

이 정보를 염두에 두고 이제 취약한 시스템에서 이 익스플로잇을 사용할 준비가 되었습니다. 우리는 다음을 할 것입니다:

  1. 익스플로잇을 사용하여 우리가 실행하고자 하는 명령이 포함된 악성 파일을 취약한 애플리케이션에 업로드합니다. 여기서 웹 서버는 이 악성 파일을 실행하여 코드를 실행합니다.
  2. 파일에는 먼저 익스플로잇이 작동했는지 확인하는 데 사용할 기본 명령이 포함됩니다.
  3. 그런 다음 취약한 시스템에 있는 파일의 내용을 읽을 것입니다.

익스플로잇을 실행하여 애플리케이션이 실행 중인 사용자의 이름을 출력합니다.

exploit.py -u http://10.10.10.10 -c "whoami"

www-data

익스플로잇을 실행하여 대상 시스템의 파일 내용 출력

exploit.py -u http://10.10.10.10 -c "cat flag.txt"
