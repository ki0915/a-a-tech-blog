1. XSS

크로스 사이트 스크립팅 공격(XSS)

XSS는 JavaScript를 기반으로 하기 때문에 언어에 대한 기본적인 이해가 있으면 도움이 됩니다. 그러나 예제 중 어느 것도 지나치게 복잡하지 않으며 클라이언트-서버 요청 및 응답에 대한 기본적인 이해도 포함합니다.

사이버 보안 커뮤니티에서 XSS로 더 잘 알려진 Cross-Site Scripting은 악성 JavaScript가 다른 사용자에 의해 실행될 의도로 웹 애플리케이션에 주입되는 주입 공격으로 분류됩니다

XSS 페이로드

페이로드란**?

XSS에서 페이로드는 대상 컴퓨터에서 실행하려는 JavaScript 코드입니다. 페이로드에는 의도와 수정의 두 부분이 있습니다.

다음은 XSS 의도의 몇 가지 예입니다.

*XSS 가능 증거**:

이것은 웹 사이트에서 XSS를 달성할 수 있음을 보여주기만 하면 되는 가장 간단한 페이로드입니다. 이는 종종 다음과 같은 텍스트 문자열이 있는 경고 상자가 페이지에 팝업되도록 하여 수행됩니다.

세션 도용**:

로그인 토큰과 같은 사용자 세션의 세부 정보는 종종 대상 시스템의 쿠키에 보관됩니다. 아래 JavaScript는 대상의 쿠키를 가져오고 base64는 쿠키를 인코딩하여 성공적인 전송을 보장한 다음 해커의 통제하에 있는 웹사이트에 게시하여 기록합니다. 해커가 이러한 쿠키를 갖게 되면 대상의 세션을 인계받아 해당 사용자로 로그인할 수 있습니다.

키 로거**:

아래 코드는 키 로거 역할을 합니다. 즉, 웹 페이지에 입력하는 모든 내용이 해커의 통제하에 있는 웹 사이트로 전달됩니다. 허용된 사용자 로그인 또는 신용 카드 세부 정보에 페이로드가 설치된 웹 사이트의 경우 매우 손상될 수 있습니다.

비즈니스 로직**:

이 페이로드는 위의 예보다 훨씬 더 구체적입니다. 이것은 특정 네트워크 리소스 또는 JavaScript 함수 호출에 관한 것입니다. 예를 들어 이라는 사용자의 이메일 주소를 변경하는 JavaScript 함수를 상상해 보십시오 user.changeEmail(). 페이로드는 다음과 같습니다.

Reflected XSS

                          Reflected XSS

Reflected XSS 는 HTTP 요청의 사용자 제공 데이터가 유효성 검사 없이 웹 페이지 소스에 포함된 경우 발생합니다.

예시 시나리오**:

잘못된 입력을 입력하면 오류 메시지가 표시되는 웹 사이트입니다. 오류 메시지의 내용은 쿼리 문자열의 오류 매개변수에서 가져오고 페이지 소스에 직접 빌드됩니다**.**

애플리케이션은 공격자가 악성 코드를 삽입할 수 있도록 허용하는 오류 매개변수 의 내용을 확인하지 않습니다 .

취약점은 아래 이미지의 시나리오에 따라 사용할 수 있습니다.

잠재적 영향**:

공격자는 JavaScript 페이로드가 포함된 다른 웹 사이트의 iframe에 링크를 보내거나 링크를 포함하여 잠재적인 피해자에게 브라우저에서 코드를 실행하도록 유도하여 잠재적으로 세션 또는 고객 정보를 노출할 수 있습니다.

Reflected XSS**를 테스트하는 방법**:

가능한 모든 진입점을 테스트해야 합니다. 여기에는 다음이 포함됩니다.

  • URL 쿼리 문자열의 매개변수
  • URL 파일 경로
  • 때때로 HTTP 헤더(실제로 악용될 가능성은 낮음)\

Stored XSS

Stored XSS

이름에서 알 수 있듯이 XSS 페이로드는 웹 애플리케이션(예: 데이터베이스)에 저장된 다음 다른 사용자가 사이트나 웹 페이지를 방문할 때 실행됩니다.

예제 시나리오**:

사용자가 댓글을 게시할 수 있는 블로그 웹사이트. 불행히도 이러한 댓글은 JavaScript를 포함하는지 또는 악성 코드를 필터링하는지 여부를 확인하지 않습니다. 이제 JavaScript가 포함된 댓글을 게시하면 데이터베이스에 저장되고 이제 기사를 방문하는 다른 모든 사용자는 브라우저에서 JavaScript를 실행하게 됩니다.

잠재적 영향**:

악의적인 JavaScript는 사용자를 다른 사이트로 리디렉션하거나, 사용자의 세션 쿠키를 훔치거나, 방문하는 사용자 역할을 하는 동안 다른 웹 사이트 작업을 수행할 수 있습니다.

저장된 XSS**를** 테스트하는 방법**:

데이터가 저장되고 다른 사용자가 액세스할 수 있는 영역에 다시 표시되는 것으로 보이는 가능한 모든 진입 지점을 테스트해야 합니다. 이들의 작은 예는 다음과 같습니다.

  • 블로그 댓글
  • 사용자 프로필 정보
  • 웹사이트 목록

때때로 개발자는 클라이언트 측에서 입력 값을 제한하는 것이 충분한 보호라고 생각하므로 웹 애플리케이션이 예상하지 못하는 것으로 값을 변경하는 것이 저장된 XSS를 발견하는 좋은 소스입니다(예: 정수를 기대하는 연령 필드). 드롭다운 메뉴 대신 악성 페이로드를 시도할 수 있는 양식을 사용하지 않고 수동으로 요청을 보냅니다. 

DOM Based XSS

DOM 기반 XSS

DOM**이란** 무엇입니까**?

DOM은 Document Object Model 의 약자 로 HTML 및 XML 문서용 프로그래밍 인터페이스입니다. 프로그램이 문서 구조, 스타일 및 내용을 변경할 수 있도록 페이지를 나타냅니다. 웹 페이지는 문서이며 이 문서는 브라우저 창에 표시되거나 HTML 소스로 표시될 수 있습니다. HTML DOM의 다이어그램이 아래에 표시됩니다.

DOM 활용

DOM 기반 XSS는 새 페이지를 로드하거나 백엔드 코드에 데이터를 제출하지 않고 브라우저에서 JavaScript 실행이 직접 발생하는 곳입니다. 실행은 웹 사이트 JavaScript 코드가 입력 또는 사용자 상호 작용에 작용할 때 발생합니다.

예제 시나리오**:

웹사이트의 JavaScript는 매개변수에서 콘텐츠를 가져온 window.location.hash 다음 현재 보고 있는 섹션의 페이지에 씁니다. 해시의 내용은 악성 코드에 대해 검사되지 않으므로 공격자가 선택한 JavaScript를 웹 페이지에 주입할 수 있습니다.

잠재적 영향**:

만들어진 링크가 잠재적 피해자에게 전송되어 다른 웹사이트로 리디렉션되거나 페이지 또는 사용자 세션에서 콘텐츠를 훔칠 수 있습니다.

Dom 기반 XSS**를** 테스트하는 방법**:

DOM 기반 XSS는 테스트하기 어려울 수 있으며 소스 코드를 읽으려면 JavaScript에 대한 어느 정도의 지식이 필요합니다. " window.location.x " 매개 변수 와 같이 공격자가 제어할 수 있는 특정 변수에 액세스하는 코드 부분을 찾아야 합니다 .

이러한 코드 조각을 찾으면 해당 코드가 어떻게 처리되는지, 값이 웹 페이지의 DOM에 기록되는지 또는 eval() 과 같은 안전하지 않은 JavaScript 메서드에 전달되는지 확인해야 합니다 .

Blind XSS

블라인드 XSS

Blind XSS는 페이로드가 다른 사용자가 볼 수 있도록 웹 사이트에 저장된다는 점에서 저장된 XSS(작업 4에서 다뤘음)와 유사하지만 이 경우 페이로드가 작동하는 것을 보거나 테스트할 수 없습니다. 먼저 자신에 대해.

예제 시나리오**:*

웹사이트에는 직원에게 메시지를 보낼 수 있는 연락처 양식이 있습니다. 공격자가 원하는 내용을 입력할 수 있도록 메시지 내용에 악성 코드가 있는지 확인되지 않습니다. 그런 다음 이러한 메시지는 직원이 개인 웹 포털에서 볼 수 있는 지원 티켓으로 전환됩니다.

잠재적 인 영향**:**

공격자의 JavaScript는 올바른 페이로드를 사용하여 공격자의 웹 사이트를 다시 호출하여 직원 포털 URL, 직원의 쿠키, 보고 있는 포털 페이지의 콘텐츠까지 노출할 수 있습니다. 이제 공격자는 잠재적으로 직원의 세션을 하이재킹하고 개인 포털에 액세스할 수 있습니다.

블라인드 XSS**를** 테스트하는 방법**:**

Blind XSS 취약성을 테스트할 때 페이로드에 콜백(일반적으로 HTTP 요청)이 있는지 확인해야 합니다. 이렇게 하면 코드가 실행되고 있는지 여부와 시기를 알 수 있습니다.

Blind XSS 공격에 널리 사용되는 도구는 xsshunter 입니다 . JavaScript로 자신만의 도구를 만드는 것이 가능하지만 이 도구는 쿠키, URL, 페이지 콘텐츠 등을 자동으로 캡처합니다.

페이로드 완성

실습

페이로드는 다른 사용자의 브라우저에서 실행하거나 웹 사이트의 취약성을 입증하기 위한 개념 증명으로 실행하려는 JavaScript 코드입니다.

페이로드에는 대상 웹 사이트에서 JavaScript를 실행할 수 있음을 증명하기 위해 JavaScript 경고 상자를 표시하는 것부터 웹 페이지 또는 사용자 세션에서 정보를 추출하는 것까지 다양한 의도가 있을 수 있습니다.
