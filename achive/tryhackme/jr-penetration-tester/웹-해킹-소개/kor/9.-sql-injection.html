1. SQL INJECTION

SQL 인젝션

주로 SQLi라고 하는 SQL(Structured Query Language) 주입은 악성 쿼리가 실행되도록 하는 웹 애플리케이션 데이터베이스 서버에 대한 공격입니다. 웹 애플리케이션이 제대로 검증되지 않은 사용자의 입력을 사용하여 데이터베이스와 통신할 때 공격자가 개인 및 고객 데이터를 도용, 삭제 또는 변경하고 웹 애플리케이션 인증 방법을 비공개로 공격할 가능성이 있습니다. 나 고객 영역 . 이것이 바로 SQLi가 가장 오래된 웹 애플리케이션 취약점 중 하나일 뿐만 아니라 가장 피해가 클 수 있는 이유입니다.

SQL 인젝션이란?

SQL 인젝션이란**?

**SQL을 사용하는 웹 애플리케이션이 SQL 주입으로 바뀔 수 있는 시점은 사용자가 제공한 데이터가 SQL 쿼리에 포함되는 시점입니다.

어떻게 생겼나요**?

온라인 블로그를 발견하고 각 블로그 항목에 고유한 ID 번호가 있는 다음 시나리오를 가정해 보십시오. 블로그 항목은 공개할 준비가 되었는지 여부에 따라 공개 또는 비공개로 설정될 수 있습니다. 각 블로그 항목의 URL은 다음과 같을 수 있습니다.

https://website.thm/blog?id=1

위의 URL에서 선택한 블로그 항목이 쿼리 문자열의 id 매개변수에서 온 것임을 알 수 있습니다. 웹 애플리케이션은 데이터베이스에서 기사를 검색해야 하며 다음과 같은 SQL 문을 사용할 수 있습니다.

SELECT * from blog where id=1 and private=0 LIMIT 1;

이전 작업에서 배운 내용을 통해 위의 SQL 문이 블로그 테이블에서 ID 번호가 1이고 개인 열이 0으로 설정된 기사를 찾고 있다는 것을 알아낼 수 있습니다. 대중이 볼 수 있으며 결과를 한 경기로 제한합니다.

이 작업의 시작 부분에서 언급했듯이 SQL 주입은 사용자 입력이 데이터베이스 쿼리에 도입될 때 도입됩니다. 이 경우 쿼리 문자열의 id 매개 변수는 SQL 쿼리에서 직접 사용됩니다.

기사 ID 2가 여전히 비공개로 잠겨 있어 웹사이트에서 볼 수 없다고 가정해 보겠습니다. 이제 대신 URL을 호출할 수 있습니다. https://website.thm/blog?id=2;--

그런 다음 SQL 문을 생성합니다.

SELECT * from blog where id=2;-- and private=0 LIMIT 1;

URL**의** 세미콜론은 SQL 문의 끝을 나타내며 두 ​​**개의** 대시는 이후의 모든 항목이 주석으로 처리되도록 합니다 . 이렇게 하면 실제로 쿼리를 실행하는 것입니다.

SELECT * from blog where id=2;--

공개로 설정되었는지 여부에 관계없이 ID가 2인 기사를 반환합니다.

인밴드 SQL 인젝션

인밴드 SQL 인젝션

In-Band SQL 주입은 탐지하고 악용하기 가장 쉬운 유형입니다. 인밴드는 웹사이트 페이지에서 SQL 인젝션 취약점을 발견한 다음 데이터베이스에서 동일한 페이지로 데이터를 추출할 수 있는 것과 같이 취약점을 악용하고 결과를 수신하는 데 사용되는 것과 동일한 통신 방법을 나타냅니다.

오류 기반 SQL 주입

이러한 유형의 SQL 주입은 데이터베이스의 오류 메시지가 브라우저 화면에 직접 출력되므로 데이터베이스 구조에 대한 정보를 쉽게 얻을 수 있는 가장 유용한 유형입니다. 이것은 종종 전체 데이터베이스를 열거하는 데 사용될 수 있습니다. 

Union 기반 SQL 삽입

이 주입 유형은 페이지에 추가 결과를 반환하기 위해 SELECT 문과 함께 SQL UNION 연산자를 활용합니다. 이 방법은 SQL 주입 취약점을 통해 대량의 데이터를 추출하는 가장 일반적인 방법입니다.
\

실습

먼저 1 UNION SELECT 1을 id 매개변수에 입력합니다.

1 UNION SELECT 1,2

같은 오류가 발생함으로 다른 열을 추가하여 반복하겠습니다.

1 UNION SELECT 1,2,3

성공, 오류 메시지가 사라지고 기사가 표시되지만 이제 기사 대신 데이터를 표시하려고 합니다. 기사가 표시되는 이유는 웹사이트 코드 어딘가에서 첫 번째로 반환된 결과를 가져와서 보여주기 때문입니다. 이 문제를 해결하려면 결과를 생성하지 않는 첫 번째 쿼리가 필요합니다. 기사 ID를 1에서 0으로 변경하면 됩니다.

0 UNION SELECT 1,2,3

이제 기사가 열 값 1, 2, 3을 반환하는 UNION 선택의 결과로 구성되어 있음을 알 수 있습니다. 반환된 값을 사용하여 더 유용한 정보를 검색할 수 있습니다. 먼저 액세스 권한이 있는 데이터베이스 이름을 가져옵니다.

0 UNION SELECT 1,2,database()

이제 이전에 숫자 3이 표시된 위치를 볼 수 있습니다. 이제 sqli_one 인 데이터베이스의 이름이 표시됩니다  .

다음 쿼리는 이 데이터베이스에 있는 테이블 목록을 수집합니다.

0 UNION SELECT 1,2,group_concat(table_name) FROM information_schema.tables WHERE table_schema = 'sqli_one'

이 쿼리에서 배워야 할 몇 가지 새로운 사항이 있습니다. 먼저**, group_concat() 메서드는   반환된 여러 행에서 지정된 열(여기서는 table_name)을 가져와 쉼표로 구분된 하나의 문자열에 넣습니다. 다음은  information_schema  데이터베이스입니다. 데이터베이스의 모든 사용자는 여기에 액세스할 수 있으며 사용자가 액세스할 수 있는 모든 데이터베이스 및 테이블에 대한 정보가 포함되어 있습니다. 이** 특정 쿼리에서 우리는 sqli_one 데이터베이스(article 및 staff_users )의 모든 테이블을 나열하는 데 관심이 있습니다   . 

첫 번째 수준은 Martin의 암호를 찾는 것을 목표로 하므로 staff_users 테이블이 우리에게 관심 있는 항목입니다. information_schema 데이터베이스를 다시 활용하여 아래 쿼리를 사용하여 이 테이블의 구조를 찾을 수 있습니다.

0 UNION SELECT 1,2,group_concat(column_name) FROM information_schema.columns WHERE table_name = 'staff_users'

이는 이전 SQL 쿼리와 유사합니다. 그러나 검색하려는 정보가 table_name에서  column_name 으로 변경되었고 information_schema 데이터베이스에서 쿼리하는 테이블이 tables에서  columns 로 변경되었으며 table_name  열의 값이  staff_users 인 행을 검색하고 있습니다  .

쿼리 결과는 staff_users 테이블에 대한 세 개의 열(id, password 및 username)을 제공합니다. 다음 쿼리에 대한 사용자 이름 및 암호 열을 사용하여 사용자 정보를 검색할 수 있습니다.

0 UNION SELECT 1,2,group_concat(username,':',password SEPARATOR '
') FROM staff_users

다시 group_concat 메서드를 사용하여 모든 행을 하나의 문자열로 반환하고 읽기 쉽게 만듭니다. 또한  ,':'**를 추가하여  사용자 이름과 비밀번호를 서로 분리했습니다. 쉼표로 구분하는 대신   읽기 쉽도록 각 결과를 별도의 줄에 강제로 표시하는 HTML 
** 태그를 선택했습니다**.**

위의 Martin의 아이디로 로그인 하면 flag 값을 얻을 수 있습니다.

블라인드 SQL 인젝션 - 인증 바이패스

블라인드 SQL 인젝션

공격 결과를 화면에서 직접 볼 수 있는 In-Band SQL 주입과 달리 블라인드 SQLi는 주입된 쿼리가 실제로 성공했는지 여부를 확인할 수 있는 피드백이 거의 또는 전혀 없는 경우입니다. 오류 메시지가 비활성화되었지만 주입은 여전히 ​​작동합니다. 전체 데이터베이스를 성공적으로 열거하기 위해 필요한 것은 약간의 피드백뿐이라는 사실에 놀랄 수도 있습니다.

인증 우회

가장 간단한 Blind SQL 주입 기술 중 하나는 로그인 양식과 같은 인증 방법을 우회하는 경우입니다. 이 경우 데이터베이스에서 데이터를 검색하는 데 관심이 없습니다. 로그인을 통과하고 싶습니다. 
사용자 데이터베이스에 연결된 로그인 양식은 종종 웹 응용 프로그램이 사용자 이름과 암호의 내용에 관심이 있는 것이 아니라 사용자 테이블에서 일치하는 쌍을 만드는지 여부에 더 관심이 있는 방식으로 개발됩니다. 기본적으로 웹 애플리케이션은 데이터베이스에 "**사용자** 이름이 bob  이고 암호가  bob123 인 사용자가 있습니까  ?"라고 묻고 데이터베이스는 예 또는 아니오(참/거짓)로 응답하고 그 대답에 따라 웹 응용 프로그램에서 진행을 허용하는지 여부. 
위의 정보를 고려하면 유효한 사용자 이름/암호 쌍을 열거할 필요가 없습니다. yes/true로 응답하는 데이터베이스 쿼리를 생성하기만 하면 됩니다.

현실적인**:

SQL 인젝션 예제의 레벨 2는 이 정확한 예제를 보여줍니다. "SQL 쿼리"라고 표시된 상자에서 데이터베이스에 대한 쿼리가 다음임을 확인할 수 있습니다.

select * from users where username='%username%' and password='%password%' LIMIT 1;

NB  %username%  및  %password%  값은 로그인 양식 필드에서 가져오며 SQL 쿼리 상자의 초기 값은 이러한 필드가 현재 비어 있으므로 비어 있습니다.

이것을 항상 true로 반환하는 쿼리로 만들려면 암호 필드에 다음을 입력할 수 있습니다.

' OR 1=1;--

그러면 SQL 쿼리가 다음과 같이 바뀝니다.

select * from users where username='' and password='' OR 1=1;

1=1은 참이고  OR  연산자를 사용했기 때문에 쿼리가 항상 참으로 반환되어 데이터베이스가 유효한 사용자 이름/암호 조합을 찾았고 액세스가 허용되어야 한다는 웹 애플리케이션 논리를 충족합니다. .

실습

' OR 1=1;--

위 구문을 비밀번호에 입력하세요.

블라인드 SQL 인젝션 – 부울 기반

부울 기반

부울 기반 SQL 주입은 참/거짓, 예/아니오, 설정/해제, 1/0 또는 두 가지 결과만 가질 수 있는 모든 응답이 될 수 있는 주입 시도에서 다시 받는 응답을 나타냅니다. 그 결과는 SQL 인젝션 페이로드가 성공했는지 여부를 확인시켜줍니다. 첫 번째 검사에서 이 제한된 응답이 많은 정보를 제공할 수 없다고 느낄 수 있습니다. 그러나 실제로는 이 두 가지 응답만으로 전체 데이터베이스 구조와 내용을 열거할 수 있습니다.

실습

https://website.thm/checkuser?username=admin

브라우저 본체에는  {"taken":true} 의 내용이 포함되어 있습니다 . 이 API 끝점은 사용자 이름이 이미 등록되어 있는지 확인하여 사용자에게 다른 사용자 이름을 선택하라는 메시지를 표시하는 많은 가입 양식에 있는 공통 기능을 복제합니다. 가져온 값이 true  로 설정되어  있으므로  사용자 이름 admin이 이미 등록되어 있다고 가정할 수 있습니다. 실제로 모의 브라우저의 주소 표시줄에서 사용자 이름을  admin 에서 admin123  으로  변경하여 이를 확인할 수 있으며 https://website.thm/checkuser?username=admin123' UNION SELECT 1,2,3 FROM information_schema.tableEnter 키를 누르면 이제 가져온 값이 false  로 변경된  것을 볼 수 있습니다  .

처리되는 SQL 쿼리는 다음과 같습니다.

select * from users where username = '%username%' LIMIT 1;

우리가 제어할 수 있는 유일한 입력은 쿼리 문자열의 사용자 이름이므로 SQL 주입을 수행하는 데 사용해야 합니다. 사용자 이름을  admin123 으로 유지 하면 여기에 추가를 시작하여 데이터베이스가 참인 것을 확인하도록 할 수 있습니다. 그러면 가져온 필드의 상태가 거짓에서 참으로 변경됩니다.

이전 수준에서와 마찬가지로 첫 번째 작업은 UNION 문을 사용하여 달성할 수 있는 사용자 테이블의 열 수를 설정하는 것입니다. 사용자 이름 값을 다음으로 변경합니다.

admin123' UNION SELECT 1;-- 

웹 애플리케이션이 false**로** 간주되는 값으로 응답했기 때문에   이것이 잘못된 열 값임을 확인할 수 있습니다. true  값을  얻을 때까지 열을 계속 추가합니다  . 사용자 이름을 아래 값으로 설정하여 답변이 3열임을 확인할 수 있습니다.

admin123' UNION SELECT 1,2,3;-- 

이제 열 수가 설정되었으므로 데이터베이스 열거 작업을 할 수 있습니다. 첫 번째 작업은 데이터베이스 이름을 찾는 것입니다. 내장된  database()  메서드를 사용한 다음  like  연산자를 사용하여 true 상태를 반환하는 결과를 찾으면 됩니다.

아래의 사용자 이름 값을 시도하고 어떤 일이 발생하는지 확인하십시오.

admin123' UNION SELECT 1,2,3 where database() like '%';--

like 연산자에는 와일드카드 값이므로 모든 항목과 일치하는 % 값만 있기 때문에 진정한 응답을 얻습니다  . 와일드카드 연산자를 a% 로 변경하면  데이터베이스 이름이 문자  a 로 시작하지 않음을 확인하는 응답이 false로 돌아가는 것을 볼 수 있습니다 . 일치하는 항목을 찾을 때까지 모든 문자, 숫자 및 - 및 _와 같은 문자를 순환할 수 있습니다. 아래를 사용자 이름 값으로 보내면   데이터베이스 이름이 문자  s 로 시작함을 확인하는 true 응답을 받게 됩니다 .

admin123' UNION SELECT 1,2,3 where database() like 's%';--

 이제 'sa%', 'sb%', 'sc%' 등과 같은 다른 실제 응답을 찾을 때까지 데이터베이스 이름의 다음 문자로 이동합니다.  sqli_three 인 데이터베이스 이름입니다  .

데이터베이스 이름을 설정했으며 이제 information_schema 데이터베이스를 활용하여 유사한 방법을 사용하여 테이블 이름을 열거하는 데 사용할 수 있습니다. 사용자 이름을 다음 값으로 설정해 보십시오.

admin123' UNION SELECT 1,2,3 FROM information_schema.tables WHERE table_schema = 'sqli_three' and table_name like 'a%';--

 이 쿼리는 데이터베이스 이름이  sqli_three 와 일치 하고 테이블 이름이 문자 a로 시작하는 테이블 테이블의 information_schema  데이터베이스  에서 결과를 찾습니다  . 위 쿼리 결과가  오답 이므로  sqli_three 데이터베이스에 a로 시작하는 테이블이 없음을 확인할 수 있습니다. 이전과 마찬가지로 긍정적인 일치를 찾을 때까지 문자, 숫자 및 문자를 순환해야 합니다.

마침내 sqli_three 데이터베이스에서 users라는 이름의 테이블을 발견하게 될 것입니다. 다음 사용자 이름 페이로드를 실행하여 확인할 수 있습니다.

admin123' UNION SELECT 1,2,3 FROM information_schema.tables WHERE table_schema = 'sqli_three' and table_name='users';--

 마지막으로 이제 로그인 자격 증명을 제대로 검색할 수 있도록 사용자 테이블 의 열 이름을 열거해야 합니다  . 다시 information_schema 데이터베이스와 이미 얻은 정보를 사용하여 열 이름에 대한 쿼리를 시작할 수 있습니다. 아래 페이로드를 사용하여   데이터베이스가 sqli_three이고 테이블 이름이 users이고 열 이름이 문자 a로 시작하는 열 테이블을 검색합니다**.**

admin123' UNION SELECT 1,2,3 FROM information_schema.COLUMNS WHERE TABLE_SCHEMA='sqli_three' and TABLE_NAME='users' and COLUMN_NAME like 'a%';

다시 한 번 일치하는 항목을 찾을 때까지 문자, 숫자 및 문자를 순환해야 합니다. 여러 결과를 찾고 있으므로 새 열 이름을 찾을 때마다 이를 페이로드에 추가해야 하므로 동일한 이름을 계속 검색하지 않아도 됩니다. 예를 들어 id 라는 열을 찾으면  원래 페이로드에 추가합니다(아래 참조).

admin123' UNION SELECT 1,2,3 FROM information_schema.COLUMNS WHERE TABLE_SCHEMA='sqli_three' and TABLE_NAME='users' and COLUMN_NAME like 'a%' and COLUMN_NAME !='id';

이 프로세스를 세 번 반복하면 열 ID, 사용자 이름 및 비밀번호를 발견할 수 있습니다. 이제 사용자  테이블에서 로그인 자격 증명을 쿼리하는 데 사용할 수 있습니다  . 먼저 아래 페이로드를 사용할 수 있는 유효한 사용자 이름을 찾아야 합니다.

admin123' UNION SELECT 1,2,3 from users where username like 'a%

모든 문자를 순환하면 사용자 이름  admin 이 있는지 확인할 수 있습니다 . 이제 사용자 이름이 있습니다. 비밀번호 찾기에 집중할 수 있습니다. 아래 페이로드는 암호를 찾는 방법을 보여줍니다.

admin123' UNION SELECT 1,2,3 from users where username='admin' and password like 'a%

모든 문자를 순환하면 암호가 3845임을 알 수 있습니다.

이제 로그인 양식에서 블라인드 SQL 삽입 취약점을 통해 열거한 사용자 이름과 비밀번호를 사용하여 다음 레벨에 액세스할 수 있습니다.

블라인드 SQL 인젝션 – 시간 기반

시간 기반

시간 기반 블라인드 SQL 주입은 동일한 요청이 전송된다는 점에서 위의 부울 기반과 매우 유사하지만 이번에는 쿼리가 잘못되었는지 또는 올바른지 시각적으로 표시하지 않습니다. 대신 올바른 쿼리에 대한 지표는 쿼리를 완료하는 데 걸리는 시간을 기반으로 합니다.  이 시간 지연은 UNION 문과 함께 SLEEP(x) 와 같은 기본 제공 메서드를 사용하여 도입됩니다  . SLEEP() 메서드는 UNION SELECT 문이 성공한 경우에만 실행됩니다. 

예를 들어 테이블의 열 수를 설정하려고 할 때 다음 쿼리를 사용합니다.

admin123' UNION SELECT SLEEP(5);--

응답 시간에 일시 중지가 없으면 쿼리가 실패한 것이므로 이전 작업과 마찬가지로 다른 열을 추가합니다.

admin123' UNION SELECT SLEEP(5),2;--

이 페이로드는 UNION 문의 성공적인 실행과 두 개의 열이 있음을 확인하는 5초의 시간 지연을 생성해야 합니다.

이제 UNION SELECT 문에 SLEEP()  메서드를  추가하여 부울 기반 SQL 주입에서 열거 프로세스를 반복할 수 있습니다   .

테이블 이름을 찾는 데 어려움을 겪고 있다면 아래 쿼리가 도움이 될 것입니다.

대역 외 SQL 인젝션

대역 외 SQL 삽입은 데이터베이스 서버에서 활성화되는 특정 기능이나 SQL 쿼리의 결과를 기반으로 일종의 외부 네트워크 호출을 만드는 웹 애플리케이션의 비즈니스 논리에 따라 달라지기 때문에 일반적이지 않습니다.

Out-Of-Band 공격은 두 개의 서로 다른 통신 채널, 즉 공격을 시작하는 채널과 결과를 수집하는 채널로 분류됩니다. 예를 들어 공격 채널은 웹 요청일 수 있고 데이터 수집 채널은 제어하는 ​​서비스에 대한 HTTP/DNS 요청을 모니터링할 수 있습니다.

1) 공격자는 인젝션 페이로드로 SQL 인젝션에 취약한 웹사이트에 요청을 한다.

2) 웹사이트는 해커의 페이로드도 전달하는 데이터베이스에 SQL 쿼리를 생성합니다.

3) 페이로드에는 데이터베이스의 데이터가 포함된 해커의 시스템으로 HTTP 요청을 강제로 보내는 요청이 포함되어 있습니다.
