1. IDOR

IDOR란?

-  IDOR은 Insecure Direct Object Reference의 약자로 액세스 제어 취약점의 일종입니다.
-   이러한 유형의 취약점은 웹 서버가 개체(파일, 데이터, 문서)를 검색하기 위해 사용자가 제공한 입력을 수신하고 입력 데이터를 너무 많이 신뢰하며 서버측에서 유효성을 확인하지 않은 경우에 발생할 수 있습니다. 

Example

- http://주소/profile?user_id=1305 와 같은 경로에서 id의 값을 다른 사람의 아이디 값으로 입력해도 제한 없이 타인의 정보를 볼 수 있는 경우

인코딩된 ID에서 IDOR 찾기

인코딩된 ID

게시물 데이터, 쿼리 문자열 또는 쿠키를 통해 페이지에서 페이지로 데이터를 전달할 때 웹 개발자는 종종 먼저 원시 데이터를 가져와 인코딩합니다. 인코딩은 수신 웹 서버가 내용을 이해할 수 있도록 합니다. 인코딩은 일반적 으로 패딩용 문자를 사용하여 이진 데이터를 ASCII 문자열로 변경합니다 . a-z, A-Z, 0-9 and =웹에서 가장 일반적인 인코딩 기술은 base64 인코딩이며 일반적으로 쉽게 찾을 수 있습니다
https://www.base64decode.org/ 와 같은 웹사이트를 사용하여 문자열을 디코딩한 다음 데이터를 편집하고 https://www.base64encode.org/를 사용하여 다시 인코딩한 다음 웹 요청을 다시 제출하여 다음을 확인할 수 있습니다.
실제로 base64 인코딩을 디코딩하여 원래 값을 알아낼 수 있습니다. 이와 같은 방법으로 IDOR을 찾아낼 수 있습니다.
그 경로로 들어가보면 되니까요.
그러니 그저 인코딩으로 경로를 숨기는 것뿐만아니라 좀 더 근본적인 방어방법이 필요합니다.

 해시된 ID

해시된 ID는 인코딩된 ID보다 다루기가 조금 더 복잡하지만 정수 값의 해시된 버전과 같이 예측 가능한 패턴을 따를 수 있습니다. 예를 들어 ID 번호 123은 md5 해싱이 사용 중인 경우 202cb962ac59075b964b07152d234b70이 됩니다.
검색된 해시를https://crackstation.net/ (값 결과에 대한 수십억 개의 해시 데이터베이스가 있음) 와 같은 웹 서비스를 통해 입력하여 일치하는 항목을 찾을 수 있는지 확인하는 것으로 해시의 원래 값을 알아낼 수 있습니다.
그러니 해시 함수를 맹신하는 것도 위험하겠죠.

 예측할 수 없는 ID

위의 방법을 사용하여 Id를 감지할 수 없는 경우 IDOR 감지의 탁월한 방법은 두 개의 계정을 만들고 두 계정 간에 Id 번호를 교환하는 것입니다. 다른 계정으로 로그인한 상태(또는 전혀 로그인하지 않은 상태)에서 ID 번호를 사용하여 다른 사용자의 콘텐츠를 볼 수 있다면 유효한 IDOR 취약점을 발견한 것입니다.

IDOR의 위치

대상으로 삼는 취약한 엔드포인트가 주소 표시줄에 표시되는 것이 아닐 수도 있습니다. 브라우저가 AJAX 요청을 통해 로드하는 콘텐츠이거나 JavaScript 파일에서 참조하는 콘텐츠일 수 있습니다. 
때때로 엔드포인트에는 개발 중에 일부 사용되어 프로덕션으로 푸시되었을 수 있는 참조되지 않은 매개 변수가 있을 수 있습니다.  예를 들어 사용자 정보(세션을 통해 인증됨)를 표시하는 /user/details 에 대한 호출을 확인할 수 있습니다  . 그러나 매개변수 마이닝으로 알려진 공격을 통해 다른 사용자의 정보를 표시하는 데 사용할 수 있는 user_id 라는 매개변수( 예: /user/details?user_id=123 )**를** 이용하여 찾을 수 도 있습니다**.**

그러니 콘텐츠와 URL 전부를 꼼꼼하게 체크해야 합니다**.**
