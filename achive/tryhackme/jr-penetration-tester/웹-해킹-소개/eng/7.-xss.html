1. XSS

Cross-Site Scripting (XSS)

XSS is based on JavaScript, so having a basic understanding of the language is helpful. However, none of the examples are overly complex and also include a basic understanding of client-server requests and responses.

Cross-Site Scripting, commonly known as XSS in the cybersecurity community, is classified as an injection attack where malicious JavaScript is injected into a web application with the intent to be executed by other users.

XSS Payload

What is a Payload?

In XSS, a payload is the JavaScript code that you want to execute on the target computer. The payload consists of two parts: the intent and the modification.

Here are a few examples of XSS intents:

Proof of XSS:

This is the simplest payload that just needs to show that XSS can be achieved on the website. This is often done by causing a popup alert box with a text string like the following:

html

코드 복사

<script>alert('XSS');</script>

Session Hijacking:

User session details, such as login tokens, are often stored in cookies on the target system. The following JavaScript retrieves the target's cookies, encodes them with base64 to ensure successful transmission, and then posts them to a website controlled by the attacker. With these cookies, the attacker can take over the target's session and log in as the user.

Key Logger:

The following code acts as a key logger, meaning all input on the webpage is sent to a website controlled by the attacker. This can be highly damaging if the payload is installed on a site where users log in or enter credit card details.

Business Logic:

This payload is much more specific than the examples above. It concerns particular network resources or JavaScript function calls. For example, imagine a JavaScript function user.changeEmail(), which changes a user's email address. The payload might look like this:

html

코드 복사

<script>user.changeEmail('attacker@hacker.thm');</script>

Reflected XSS

Reflected XSS occurs when user-supplied data in an HTTP request is included in the web page source without validation.

Example Scenario:

A website that displays an error message when invalid input is entered. The error message content is pulled from the error parameter in the query string and is directly included in the page source.

Potential Impact:

The attacker can send a link with a JavaScript payload included in an iframe from another website, tricking the victim into executing code in their browser and potentially exposing session or customer information.

Testing for Reflected XSS:

You should test all possible entry points, including:

  • URL query string parameters
  • URL file paths
  • Sometimes HTTP headers (although the risk is lower in practice)

Stored XSS

Stored XSS

As the name suggests, the XSS payload is stored in the web application (e.g., database) and is then executed when other users visit the site or webpage.

Example Scenario:

A blog website where users can post comments. Unfortunately, these comments are not checked to see if they contain JavaScript or malicious code. Posting a comment with JavaScript will store it in the database, and now every user who visits the article will execute JavaScript in their browser.

Potential Impact:

Malicious JavaScript can redirect users to another site, steal users' session cookies, or perform other actions on the web site while impersonating the visiting user.

Testing for Stored XSS:

Test all possible entry points where data is stored and then re-displayed to other users. Examples include:

  • Blog comments
  • User profile information
  • Website listings

Sometimes developers mistakenly believe that limiting input values on the client side is sufficient protection, so testing stored XSS can involve changing values unexpectedly (e.g., trying a malicious payload in a field that expects an integer). Test forms manually by sending requests without using dropdown menus.

DOM Based XSS

What is DOM-Based XSS?

DOM stands for Document Object Model, a programming interface for HTML and XML documents. It represents the structure, style, and content of a document so programs can modify it. A web page is a document, and this document can be displayed in a browser window or as HTML source. An HTML DOM diagram is shown below.

How DOM is Used

DOM-based XSS occurs when JavaScript execution happens directly in the browser without loading a new page or submitting data to the backend code. This execution occurs when web page JavaScript interacts with input or user actions.

Example Scenario:

A website’s JavaScript writes content from the window.location.hash parameter into the current page section. Since the hash content is not checked for malicious code, an attacker can inject JavaScript into the webpage.

Potential Impact:

An attacker can send a crafted link to a potential victim, redirecting them to another website or stealing content from the page or user session.

Testing for DOM-Based XSS:

Testing for DOM-based XSS can be challenging and requires some knowledge of JavaScript. Look for specific code sections that handle variables attackers can control, such as "window.location.x" parameters.

Once you find such code snippets, you should check how they are processed, whether values are recorded in the page's DOM, or if they are passed to unsafe JavaScript methods like eval().

Blind XSS

Blind XSS

Blind XSS is similar to stored XSS in that the payload is stored in the website for others to see, but in this case, the attacker cannot see or test the payload’s execution themselves.

Example Scenario:

A website with a contact form that allows users to send messages. The message content is not checked for malicious code. These messages are then converted into support tickets visible in a staff member’s personal web portal.

Potential Impact:

An attacker’s JavaScript can use a correct payload to call back to the attacker’s website, exposing the staff portal URL, staff cookies, and the contents of the portal page. The attacker could potentially hijack the staff session and access the personal portal.

Testing for Blind XSS:

When testing for Blind XSS vulnerabilities, ensure that the payload includes a callback (typically an HTTP request) so you can determine if and when the code is executed.

A widely used tool for Blind XSS attacks is xsshunter. While it is possible to create your own tool with JavaScript, this tool automatically captures cookies, URLs, and page content.

Completing Payloads

Practical Exercise

A payload is JavaScript code designed to execute in another user's browser or to prove the vulnerability of a website.

Payloads can range from showing a JavaScript alert box to extracting information from the web page or user session.

The type of payload used depends on how the JavaScript payload is reflected in the target website's code. To demonstrate this, click the green start machine button on the right, and when the machine loads, open the link in a new tab.

Each level's goal is to execute a JavaScript alert function using the string THM. For example:

html

코드 복사

<script>alert('THM');</script>

Level 1:

A form appears asking for a name, and when you enter a name, it is displayed below. For example:

html

코드 복사

<script>alert('THM');</script>

When you click Enter, a popup alert with the string THM will appear, and the page source will reflect the result.

Level 2:

Similar to the previous level, a prompt to enter a name is shown. This time, the name is reflected inside the input tag.

html

코드 복사

"><script>alert('THM');</script>

The important part of the payload is closing the input tag with "> and then closing the tag. This will allow the JavaScript payload to execute properly.

Level 3:

A form asking for a name appears, and the name is reflected inside a textarea tag. Use the following payload to escape the textarea tag:

html

코드 복사

</textarea><script>alert('THM');</script>

The important part of this payload is closing the textarea element with </textarea>, so the script can be executed.

Level 4:

The form asks for a name, which appears to be similar to Level 1. However, the page source shows that the name is reflected inside some JavaScript code.

html

코드 복사

';alert('THM');//

The payload ends the current command with ;, and // makes everything after it a comment. This way, the JavaScript code will be executed.

Level 5:

This level looks similar to Level 1, but the payload doesn’t work as expected due to a filter removing potentially dangerous words like <script>.

Original Payload:

html

코드 복사

<sscriptcript>alert('THM');</sscriptcript>

Filtered Text:

html

코드 복사

<script>alert('THM');</script>

The payload uses <sscriptcript> which is filtered and then executes as <script>.

Level 6:

This level requires escaping from the input tag value attribute, similar to Level 2 but with different results. Check the page source to determine why the payload doesn’t work.
