1. IDOR

What is IDOR?

  • IDOR stands for Insecure Direct Object Reference and is a type of access control vulnerability.
  • This type of vulnerability occurs when a web server receives input from a user to search for an object (file, data, document) and trusts this input too much without validating it on the server side.
Example
  • For example, if you can input a different person's ID value in a path like http://address/profile?user_id=1305 and view others' information without restriction.

Finding IDOR in Encoded IDs

Encoded IDs

When transferring data between pages via post data, query strings, or cookies, web developers often encode raw data first. Encoding allows the receiving web server to understand the content. Encoding typically converts binary data to an ASCII string using padding characters such as a-z, A-Z, 0-9, and =. The most common encoding technique on the web is base64 encoding, which is generally easy to find.

You can decode a string using websites like https://www.base64decode.org/, then edit the data and re-encode it using https://www.base64encode.org/, and resubmit the web request to verify changes.

Base64 encoding can indeed be decoded to reveal the original value. By finding and examining the path using this method, you can detect IDOR vulnerabilities.

However, simply hiding paths with encoding isn't enough; more fundamental defenses are needed.

Finding IDOR in Hashed IDs

Hashed IDs are a bit more complex than encoded IDs, but they may follow predictable patterns, like a hashed version of an integer value. For example, if ID number 123 is hashed using md5, it becomes 202cb962ac59075b964b07152d234b70.

You can find the original value of a hash by using web services like https://crackstation.net/ that have billions of hash databases. By inputting the hash, you can see if there are matching entries.

Thus, relying solely on hash functions can be risky.

Finding IDOR in Unpredictable IDs

Unpredictable IDs

If you can't detect IDs using the methods above, an excellent way to detect IDOR is to create two accounts and exchange IDs between the two accounts. If you can use the ID number from one account to view content belonging to another user while logged in as a different account (or while not logged in at all), you've found a valid IDOR vulnerability.

Location of IDOR

The vulnerable endpoint may not always be visible in the address bar. It could be content loaded through AJAX requests or referenced in JavaScript files. Sometimes, endpoints might have unused parameters pushed to production that were used during development. For example, you might see a call to /user/details that shows user information authenticated through a session. However, through parameter mining, you might find a parameter called user_id (e.g., /user/details?user_id=123) that can be used to display information for other users.

So, you must thoroughly check all content and URLs.

Practical Example of IDOR

Practical Exercise

First, access the website provided by the site and create an account. After logging in, click on the account tab.

After this screen appears and you refresh, you will see a call to the endpoint /api/v1/customer?id= in the developer tools.

This page returns user ID, username, and email address in JSON format. The displayed user information is fetched from the ID parameter in the query string.

Check the "customer" here. You can inspect the parameter and test for vulnerabilities.
