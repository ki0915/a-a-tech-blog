타당성 검토

  • 프로젝트의 성공 확률을 높여주기 위해서 수행해야 한다.
    - 기대효과, 개발 비용, 위험 분석, 품질 관리 검토, 수행
  • 주기적으로 검토해야한다.

요구 사항 정의(기본 설계, 기본 분석)

  • 기본 설계: 사용자의 요구 사랑을 정의하고 분석(사용자 경영진 참여)
  • 시스템 요구 사항을 공식적으로 문서화한다.
    - 내장 감사 모델(eam)이 필요한지 검토, 필요하면 개념 설계 포함한다.
  • 사업사례(business case) 작성 단계
    - 사용자 인수 테스트 사양서 검토
  • 인수 테스트 요구사항이 이 단계에서 정해진다.
  • [정보 시스템 구입 개발 및 관리 2 > ^34c1ec]

설계(상세 설계)

  • 상세 설계
    - cisa가 집중적으로 확인
  • 프로그램이 적절한 통제 절차로 프로그래밍 되어 있는지 확인한다.
    - 모든 변경 사항에 경영진 승인이 있었는지 확인한다.
  • 기준선: 범위 변경 문제를 완화할 수 있다.
  • 기준선은  형상 관리의 기준, 변경 관리의 대상이 된다.

개발

  • 소스코드 작성 표준이 중요한 표준이 된다. 
  • 프로그래밍 과정에서 오류가 적절히 해결되었는지 확인한다.

CISA 테스트

단위 테스트

  • 개별 모듈이나 프로그램을 테스트한다. 프로그램 기능이 사양서대로인지 테스트

통합 테스트

  • 컴포넌트 간의 연결과 통합을 테스트

시스템 테스트

  • 여러 모듈과 기능을 모아 하나의 시스템으로 테스트
종류

복구 테스트: 복구가 잘 되는지 테스트
보안 테스트: 접근 통제 잘 되었나 보안 취약점 없나 테스트
스트레스 및 볼륨 테스트: 부하나 한계 테스트
성능 테스트: 성능을 테스트

인수 테스트

- 사용자 요구사항대로 구현되었는지 테스트(최종 사용자가 검증)

  • 완료하면 사용자가 인수증에 서명

기타 테스트

종류

알파 테스트: 조직 내 사용자 조직이 테스트
베타 테스트: 조직 외부의 한정된 사용자들에게 테스트(보안 취약)
화이트 박스 테스트: 프로그램의 로직을 검사한다.
블랙 박스 테스트: 프로그램의 외부 기능을 테스트한다.
병행 테스트: 현재 시스템과 새로운 시스템이 결과를 비교
ITF(통합 설비 시설): 현재 운영 중인 컴퓨터에 테스트 데이터를 입력하고 결과와 기대값을 비교
회귀 테스트: 새로운 프로그램 도입이나 업데이트 때문에 문제가 생기지 않을 거라는 걸 테스트
사교성 테스트: 새로운 프로그램이나 시스템이 도입되었을 때 문제가 없을 지 테스트

상향식 테스트

  • 작은 모듈에서 시스템 전체까지 테스트하는 방식이다.
    - 초기 모듈 오류 발견 쉽다.
  • 구현하기 쉽다.

하향식 테스트

  • 커다란 부분에서 작은 부분으로 하양식으로 진행
    - 인터페이스 오류 발견 쉽다.
  • 기능이나 중요 처리를 검사
  • cisa에서 구현을 개발이 아니라 운영으로 이관이다.
