Information System Acquisition, Development, and Management**

Key Concepts

Project Charter

  • Officially authorizes the existence of a project.
  • Grants the project manager the authority to use organizational resources.
  • Approved by a senior executive or sponsor.

Business Case

  • Important for exams
  • Conducted to address organizational strategic objectives or business issues.
  • Essentially answers the question: Why is this project being undertaken?

Key Issues

  1. Necessity: Why is this project needed?
  2. Legitimacy: Is this project legal and legitimate?
  3. Expected Benefits: What benefits will this project bring to the organization?
Tip
  • The business case should be included in the project charter.
  • The business case is performed during the feasibility study.
  • It may be reviewed and modified later.
  • It must be approved by senior management or sponsors.
  • Any changes require approval from senior management.

Project Steering Committee

  • A consultative body consisting of senior representatives.
  • Chairperson: Project Sponsor (Project Owner)
  • Provides strategic direction for the project, represents the interests of various groups, and holds full responsibility.
Tip
  • All responsibility for the project lies with the Steering Committee.
  • Holds the authority for corrective actions and project termination.
  • With great responsibility comes great authority.

Committees

Audit Committee (Composed of external directors not involved in management)

  • Symbolizes the independence and authority of the audit function.
  • Thus, extensive discussions are held with the Audit Committee to ensure independence during the audit process.

IT Strategy Committee

  • Provides strategic direction and advice to ensure the organization's strategy aligns with its IS (IT) strategy.

IS Operations Committee

  • Ensures that "project" activities align with the organization's information strategy objectives, providing strategic direction and advice.

Post-Implementation Review

  • A review conducted 6 to 12 months after project completion to assess whether the project has met its objectives.
Tip
  • Performed by the end-users and the project development team.
  • Different from a post-project review.

Project Cost Estimation Techniques

Important for exams

Source Lines of Code (SLOC)

  • Estimates project size based on the number of lines of code.

    Function Point Analysis (FPA)

  • Estimates project size based on the size of each function.
  • The most effective and preferred technique.

    Delphi Technique

  • Gathers expert opinions anonymously.

    Regression Analysis

  • Uses historical data for analysis.

Project Scheduling

PERT (Program Evaluation and Review Technique)

  • Views the project as a set of related activities and expresses their relationships.
  • Path: A sequence of tasks without loops.
  • Shortest Project Completion Time: The longest path.

CPM (Critical Path Method)

  • Uses the average of the shortest and longest completion times.
  • Slack Time: The difference between the minimum and maximum completion times.
  • Allocating more resources to the critical path (CP) can shorten project completion time.

Gantt Chart

  • Displays time on the horizontal axis and project activities on the vertical axis, using bars to represent tasks.

Time Boxing

  • Requires delivering results within a set time frame.
  • Allows defining outcomes with limited time and resources.

Milestone

  • Marks important events or points during the project. -> These points must be reached for the project to continue.
  • Used for schedule management.

Application System Development and Procurement Life Cycle

Waterfall Model

  • Progresses sequentially, emphasizing strict project management and standardization.
  • Used when the project is stable, and requirements do not change significantly. Important for exams

Prototyping Model

  • An interim model shaped according to user requirements.
    Types
  • Experimental Prototyping: Created only for communication with users, then discarded.
  • Evolutionary Prototyping: Created for communication and as an initial version.

    Spiral Model

  • Combines evolutionary prototyping with risk management.
  • Involves iterative cycles of planning, risk analysis, development, and customer evaluation.
  • Focuses on risk management.

    Iterative Model

  • A combination of Waterfall, Prototyping, and Spiral models.
  • All phases are repeated iteratively.
